# Blog Review Skill

블로그 포스트를 3단계로 리뷰하고 발견된 문제를 즉시 수정한다.

---

## 핵심 원칙

- 3단계 순서를 지킨다: 엔지니어링 정확성 → 언어 품질 → 포맷 일관성
- 각 단계 완료 후 즉시 Edit으로 수정 적용한다. 마지막에 한 번에 몰아서 수정하지 않는다.
- 기술적 사실 오류는 근거를 명시한 뒤 수정한다. 확신이 없으면 수정하지 않고 플래그만 남긴다.
- 톤 위반(격식체 사용)은 반드시 수정한다.

---

## 워크플로우

### Step 1 — 파일 확인

대화에서 `@file` 참조 또는 경로가 명시된 경우 해당 파일을 Read한다.
경로가 없으면:

```
Glob: content/posts/**/*.md  →  수정 시각 기준 최신 파일 1개 선택 후 사용자에게 확인
```

파일 경로를 확인한 뒤 **전체 내용을 한 번에 Read**한다.

---

### Step 2 — Phase 1: 엔지니어링 내용 검증

기술적 정확성을 검토한다. 세부 체크리스트는 `references/review-checklist.md` 참조.

검토 항목:
- **사실 오류**: 잘못된 기술 개념, 틀린 동작 원리 설명
- **명칭 오류**: Kubernetes API 오브젝트명, 필드명, 시스템 콜명 오타
- **경로/명령 오류**: 파일 경로, CLI 플래그, YAML 필드 오타
- **코드 블록 오류**: YAML/bash 예시의 문법·필드 정확성
- **내부 일관성**: 글 안에서 동일 개념을 다르게 설명하는 경우

발견된 문제를 Edit으로 즉시 수정한다.
확신이 없는 항목은 수정하지 않고 리포트에 `⚠️ 검토 필요` 로 표시한다.

---

### Step 3 — Phase 2: 맞춤법 · 띄어쓰기 · 톤

언어 품질을 검토한다.

**맞춤법 · 표현 오류 (자주 등장하는 패턴):**

| 잘못된 표현 | 올바른 표현 |
|------------|------------|
| 가르키다 | 가리키다 |
| ~이/가 아니라 → 문맥상 틀림 | 상황에 맞게 수정 |
| 돼 / 되 혼용 | 문맥 판단 후 수정 |
| 않 / 안 혼용 | 문맥 판단 후 수정 |
| ~로서 / ~로써 혼용 | 문맥 판단 후 수정 |
| 포할하다 | 포함하다 |
| ~서만 (조사 누락) | ~에서만 |

**띄어쓰기:**
- 영어 단어 앞뒤 한글과의 공백: `API서버` → `API 서버`
- 숫자와 단위 사이 공백: `1MiB` 는 붙여쓰기 허용, `1 MiB` 도 허용 — 글 내 일관성 유지
- 조사는 앞 단어에 붙임: `~을 통해` (O), `~을통해` (X)
- `조금더` → `조금 더`, `훨씬더` → `훨씬 더`

**문장 부호:**
- 문장 끝에 마침표 누락 (단, 제목·목록 항목은 마침표 생략 허용)
- 쉼표 없이 이어지는 긴 문장 분리 검토

**톤 일관성 (MUST):**
- 비격식체(반말) 유지: `~입니다` / `~합니다` / `~하세요` / `~겠습니다` / `~줍니다` → 즉시 수정
- 예: `사용합니다` → `사용한다`, `알아보겠습니다` → `알아보자`
- 예: `있습니다` → `있다`, `됩니다` → `된다`
- 예: `해줍니다` → `해준다`, `설정해 줍니다` → `설정해 준다`
- 예: `영역입니다` → `영역이다`, `것입니다` → `것이다`, `필요합니다` → `필요하다`

발견된 문제를 Edit으로 즉시 수정한다.

---

### Step 4 — Phase 3: Markdown 포맷 일관성

**Frontmatter:**

| 필드 | 조건 |
|------|------|
| `title` | 존재 · 비어있지 않음 |
| `description` | 존재 · 1줄 이상 · 의미있는 설명 (오타 없음) |
| `date` | `YYYY-MM-DD` 형식 |
| `status` | `deploy` 또는 `writing` |
| `thumbnail` | `./파일명` 형식 존재 |
| `tags` | 1개 이상 |

**헤딩 계층:**
- h1 → h2 → h3 순서를 지킨다. 레벨 건너뜀 금지 (h1 다음 h3 등)
- h1은 파일 당 1개 권장
- **bold 단독 문단을 헤딩 대용으로 사용하는 패턴 검출**: `**소제목**`이 빈 줄로 둘러싸인 단독 문단으로 등장할 때, 헤딩 계층상 소제목 역할이면 적절한 `####` 헤딩으로 교체

**코드 블록:**
- 모든 코드 블록에 언어 지정 필수: ` ```yaml `, ` ```bash `, ` ```go ` 등
- 언어 미지정 블록 발견 시 내용을 보고 판단하여 추가

**목록 스타일:**
- 같은 섹션 내에서 `-` 와 `*` 혼용 금지 — 글 전체 기준으로 다수 스타일을 기준으로 통일

**정의 목록 마커 오용:**
- 목록 항목(`- **제목**`) 다음 줄에 `: 설명` 패턴이 오면 일부 Markdown 파서가 definition list로 해석해 렌더링이 깨짐
- 수정 방법: 항목 헤더에 인라인으로 통합 → `- **제목**: 설명`

**유니코드 곱슬따옴표 + 이탤릭 충돌:**
- `*'텍스트'*` 패턴 — 곱슬따옴표(U+2018 `'`, U+2019 `'`)가 이탤릭 마커 `*` 바로 안쪽에 오면 CommonMark 파서가 이탤릭으로 인식하지 못하고 `*`가 그대로 노출됨
- 수정 방법: 곱슬따옴표 제거 → `*텍스트*`

**Non-breaking space (`\xa0`):**
- 노션·웹에서 복붙한 텍스트에 일반 공백 대신 `\xa0`이 섞이는 경우가 있음
- Markdown 파싱과 편집 도구 오작동의 원인이 될 수 있으므로 발견 시 일반 공백으로 교체

**이미지:**
- `![](path)` 형식에서 alt text 누락 시 파일명 기반으로 추가

발견된 문제를 Edit으로 즉시 수정한다.

---

### Step 5 — 리뷰 리포트 출력

모든 수정이 완료된 뒤 아래 형식으로 요약한다.
수정이 없는 카테고리는 "✅ 이상 없음"으로 표시한다.

```
## 리뷰 완료: {파일 경로}

### Phase 1 — 엔지니어링 검증
- ✅ 이상 없음  (또는 수정 목록)

### Phase 2 — 맞춤법/톤
| 위치 | 수정 전 | 수정 후 |
|------|--------|--------|
| ...  | ...    | ...    |

### Phase 3 — Markdown 포맷
- ✅ 이상 없음  (또는 수정 목록)

---
총 N건 수정 완료.  ⚠️ 검토 필요 M건 (위 표시 항목 확인 요망)
```

---

## 주의사항

- Phase 1 기술 사실 수정은 확신 있을 때만 적용한다. 도메인 전문 지식(Kubernetes, Linux kernel, OS 원리)이 필요한 경우 신중하게 판단한다.
- 글쓴이의 의도적인 문체나 강조 표현을 과도하게 수정하지 않는다.
- 코드 블록 내부 내용은 맞춤법/띄어쓰기 수정 대상에서 제외한다.

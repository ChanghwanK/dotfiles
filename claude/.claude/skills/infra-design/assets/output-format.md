# infra-design 출력 템플릿

구분 기호는 콜론·쉼표·괄호를 쓴다 (em dash 금지). 격식체(~입니다, ~합니다)를 사용한다.

```markdown
# {설계 주제} 설계 제안

## Summary
- **무엇을**: (설계 대상과 목표 한 줄)
- **추천**: Option {X} ({조건부 추천 한 줄: 우리 상황이 어느 조건에 해당하는지})
- **주의**: (one-way door 여부 또는 최대 리스크 한 줄)

## 현재 인프라 컨텍스트
(context-collector 결과 기반. 실제 확인 못 한 항목은 "(미확인)" 표기)
- 관련 기존 구성: ...
- 관련 ADR/기존 결정: ... (설계와 충돌 시 명시)
- 가드레일 제약: ...

## 요구사항 정리
- 목표/동인: ...
- 제약: 규모 / 환경 / 예산 / 마감

## 설계 옵션

### Option A: {옵션명}
- **아키텍처 개요**: (구성요소와 연결 관계. 필요 시 텍스트 다이어그램)
- **핵심 구성**: (limit·quota·사이징 근거 포함) [n]
- **우리 인프라 접점**: (재사용 컴포넌트, 신규 도입 컴포넌트)

### Option B: {옵션명}
(동일 구조)

## 트레이드오프 분석

| 축 | Option A | Option B |
|----|----------|----------|
| 성능 | | |
| 비용 (월 추정) | | |
| 운영 부담 | | |
| 복잡도 | | |
| 위험도 | | |

- **2차 효과 (6개월 후)**:
  - Option A: ... (부채 증가율: 선형/복리)
  - Option B: ...
- **one-way door**: (되돌리기 어려운 결정 명시, 없으면 "없음")
- **조건부 추천**: X 상황이면 A, Y 상황이면 B. 우리 인프라는 {판단 근거}이므로 {추천}.
- **ROI Self-Check** (신규 도입 시): 비용 / 단순성(Managed 대안) / 운영 부담 / 기술 부채 4항 점검 결과

## 추천 설계 상세
(추천 옵션의 구체 설계: 단계, 구성값, 마이그레이션 경로)

## 위험과 롤백
- **Blast radius**: ...
- **실패 시나리오**: ...
- **롤백 방법**: ... (소요 시간 포함)

## References
[1] {제목} · {URL} · T1 · {날짜 또는 대상 버전}
[2] {제목} · {URL} · T2 · {날짜}
```

## 규칙

- 모든 설계 근거 주장에 인라인 `[n]` 인용. References에 T3 소스 금지
- 비용 수치는 단가 출처(aws-pricing MCP 조회 / 공식 pricing 페이지 / 추정)와 수량 근거를 병기. "추정"은 단가 확인 실패 시 최후 수단이며 산정 가정을 함께 적는다
- 트레이드오프 표의 각 셀은 한 줄 요약 + 필요 시 인용
- 옵션이 3개 이상이면 표 열을 늘린다. 비교 축 5개는 고정(생략 금지)

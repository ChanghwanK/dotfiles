# task:review: 작업 후 성과 측정, PAR 작성 & 성장 회고 스킬

작업 완료 후 대화 맥락 전체를 분석하여 세 가지를 수행한다.

- **Part A: 성과 측정**: 최초 목표 대비 무엇을 달성했는가를 증거 기반으로 측정한다. (사실 확정)
- **Part A.5: PAR 성과 문장**: 측정된 성과를 Problem / Action / Result 구조의 이력서·평가용 문장으로 변환한다. (전환)
- **Part B: 성장 회고**: 다음 번에 더 잘 하기 위한 구체적 인사이트를 생성한다. (성찰)

**순서가 중요하다.** 성과 측정(사실)을 먼저 확정한 뒤 PAR 문장(전환), 성장 회고(성찰)로 넘어간다.
"무엇을 달성했는가"가 확정돼야 "어떻게 표현할 것인가"와 "어떻게 더 잘 할 수 있었는가"가 의미를 가진다.

**실행 범위:** 기본은 Part A + Part A.5 + Part B 모두 수행한다.
사용자가 "성과만" 요청하면 Part A만, "회고만" 요청하면 Part B만 출력한다.
사용자가 "PAR만" 또는 "이력서 문장만" 요청해도 Step 1~4의 최소 근거 추출은 내부적으로 수행하고, 출력만 Part A.5로 제한한다.

---

## 핵심 원칙

**공통**
- **증거 기반**: 대화에서 실제 발생한 일만 기반으로 분석. 추측 금지.
- **작업 타입 인식**: RCA / 배포 / 설계 / 개발 / 트러블슈팅별 차원이 다르다.

**Part A (성과 측정) 전용**
- **목표 baseline 우선**: 성과는 반드시 "최초 목표 대비"로 측정한다. 목표 식별이 선행돼야 한다.
- **정량 우선**: before→after 수치를 우선한다. 불가능할 때만 정성으로.
- **산출물과 효과 분리**: PR/commit(산출물)과 알럿 해소/비용 절감(효과)을 구분한다. 효과가 진짜 성과다.
- **정직한 판정**: 미달성을 달성으로 포장하지 않는다. 부분 달성은 부분으로, 미달은 미달로.

**Part A.5 (PAR 성과 문장) 전용**
- **Problem / Action / Result 분리**: 문제, 실행, 결과가 각각 보여야 한다. 기술명 나열로 끝내지 않는다.
- **Problem은 Situation 포함**: Problem은 `Situation + Pain/Risk/Goal`로 정의한다. 어떤 맥락에서 왜 중요한 문제였는지 함께 드러낸다.
- **Result 우선 검증**: Result는 Step 2의 측정된 효과만 사용한다. 증거 없는 임팩트는 넣지 않는다.
- **역할 명확화**: "개선됨"보다 "무엇을 설계/자동화/표준화/검증했는지"를 주어가 보이게 쓴다.
- **사용처별 톤 조정**: 이력서는 간결한 성과 bullet, 경력기술서/성과평가는 맥락과 판단 근거를 조금 더 남긴다.
- **이력서 bullet은 표준 포맷 준수**: `~/.claude/docs/resume-format-convention.md` 기준(명사형 종결,
  Before/After 정량, 대괄호·화살표 표기)을 따른다. 경력기술서/성과평가용은 이 표준 대상이 아니며 기존
  PAR 서술형을 그대로 유지한다.

**Part B (성장 회고) 전용**
- **비판보다 성장**: 무엇이 나빴는지가 아니라 무엇을 달리 했으면 결과가 달라졌는지.
- **구체성**: "더 잘 확인해야 했다" 금지. "어떤 명령어로 무엇을 확인했어야 했는지" 명시.
- **학습 연결**: 놓친 것은 반드시 "왜 몰랐는지" 또는 "어떤 개념을 알면 됐는지"로 연결.

## Step 0: 작업 타입 감지

대화 맥락에서 이번 작업의 주 타입을 결정한다. 복수 타입이면 주 타입 1개 + 부 타입 병기.

| 타입 | 감지 신호 |
|------|-----------|
| **RCA** | 알럿 URL 분석, kubectl 조사, 메트릭 쿼리, 5 Whys, root cause 탐색 |
| **Deployment** | image tag 업데이트, PR 생성, ArgoCD sync, canary, rollout |
| **Design** | Tech Spec, 아키텍처 논의, 트레이드오프 비교, ADR |
| **Development** | 코드 수정, 버그 수정, 기능 추가, 리팩터링 |
| **Troubleshooting** | 특정 에러 해결, 설정 수정, 디버깅 (RCA보다 범위 작음) |
| **Operations** | 인프라 변경, 업그레이드, 정리 작업, 모니터링 설정 |

# Part A: 성과 측정 (Outcome Measurement)

## Step 1: 작업 분석 & 최초 목표 식별

대화를 처음부터 끝까지 스캔하여 "무엇을 하려 했고 무엇을 했는가"를 확정한다.
성과 측정의 baseline은 **최초 목표**이므로 이 단계가 가장 중요하다.

**식별 항목:**
- **최초 목표(명시적)**: 사용자가 대화 시작 시 요청한 것. 원문 의도 그대로.
- **실질 목표(암묵적)**: 작업 중 드러난 진짜 목표. 사용자가 X를 요청했어도 근본 문제가 Y였다면 Y를 명시.
- **수행한 작업**: 실제로 실행된 핵심 작업 (1-3줄).
- **범위 변화**: 작업 중 목표가 추가/축소/전환됐는가 (scope creep / pivot). 없으면 "없음".

```
**작업 타입**: {주 타입} (+ {부 타입})
**최초 목표(명시적)**: {사용자 최초 요청}
**실질 목표(암묵적)**: {작업 중 드러난 진짜 목표: 명시적과 같으면 "동일"}
**수행한 작업**: {실제로 한 것 1-3줄}
**범위 변화**: {추가/축소/전환된 것: 없으면 "없음"}
```

---

## Step 2: 성과 측정

실제로 산출/달성된 것을 측정한다. **주장 금지, 증거 필수**: 대화에서 확인된 산출물/메트릭만.

**측정 차원** (팀 가중치 순: 생산성 > 비용 > 안정성):

| 차원 | 측정 대상 예시 |
|------|---------------|
| **생산성** | 자동화, 병목 해소, 작업 시간 단축, self-service 가능해진 것 |
| **비용** | 리소스/네트워크 비용 절감 (정량 금액 또는 용량) |
| **안정성** | 알럿 해소, 장애 복구, 가용성/SLO 개선 |
| **지식/부채** | 문서화, 런북, 기술 부채 해소 (또는 의도적으로 생성한 부채) |

**산출물 vs 효과를 분리한다:**
- **산출물(Artifacts)**: PR/commit/문서/설정 변경: 무엇을 만들었는가
- **효과(Outcomes)**: 알럿 해소, 88%→41%, 145GB 회수: 무엇이 실제로 달라졌는가

각 효과는 정량(before→after)을 우선하고, 정량 불가 시 정성으로 명시한다.

```
**산출물**
- {PR/commit/문서 등: 식별자 포함}

**측정된 효과**
- [생산성] {달라진 것}: {수치 또는 정성}: 근거: {대화 근거}
- [비용] {달라진 것}: {수치 또는 정성}: 근거: {대화 근거}
- [안정성] {달라진 것}: {수치 또는 정성}: 근거: {대화 근거}
- [지식/부채] {달라진 것}: {수치 또는 정성}: 근거: {대화 근거}
```

해당 차원의 효과가 없으면 항목을 생략한다.

## Step 3: 목표 달성 여부 판정

Step 1의 목표와 Step 2의 성과를 대조하여 판정한다.

**판정 등급:**

| 등급 | 의미 |
|------|------|
| **달성** | 목표 100% 충족 |
| **부분 달성** | 핵심은 충족, 일부 미완 (미완 항목 + 이유 명시) |
| **미달성** | 목표 미충족 (이유 + blocker 명시) |
| **초과 달성** | 목표 외 추가 가치까지 산출 |

규칙:
- 목표가 복수면 **항목별로** 판정한다.
- 측정 기준은 **실질 목표(암묵적)** 우선. 명시적 목표만 달성하고 근본 문제를 못 풀었다면 부분 달성이다.
- **계획 외 결과(side effect)**를 반드시 별도 기록한다: 긍정(추가 발견·개선)과 부정(새로 생긴 부채·리스크) 모두.

```
**목표별 판정**
- {목표1}: **달성**: {성과 증거}
- {목표2}: **부분 달성**: {부분 성과} / 잔여: {남은 것 + 이유}
- {목표3}: **미달성**: {이유 + blocker}

**계획 외 결과**
- 긍정: {side benefit: 없으면 "없음"}
- 부정: {새로 생긴 리스크/부채: 없으면 "없음"}
```

Part A 전체를 압축한다.

```
**전체 달성도**: {종합 판정: 예: 핵심 목표 달성, 후속 1건 잔여}

```

# Part A.5: PAR 성과 문장 (Resume / Review Writing)

## Step 4.5: PAR 성과 문장 작성

Step 1~4에서 확인된 목표, 실행, 효과를 이력서·성과평가에 바로 쓸 수 있는 문장으로 변환한다.
PAR은 성과를 과장하기 위한 포맷이 아니라, 문제 해결의 맥락과 실제 임팩트를 압축하는 포맷이다.

**PAR 구성:**
- **Problem**: `Situation + Pain/Risk/Goal`. 어떤 맥락에서 어떤 문제가 있었고, 왜 해결이 필요했는가
  - Situation: 어떤 운영/비즈니스/개발 맥락이었는가
  - Pain/Risk: 그 상황에서 무엇이 병목·위험·비효율이었는가
  - Goal: 그래서 무엇을 개선해야 했는가
- **Action**: 내가 어떤 판단으로 무엇을 실행했는가
- **Result**: 그 결과 무엇이 측정 가능하게 달라졌는가

**작성 규칙:**
- Problem은 "기술을 썼다"가 아니라 "어떤 상황에서 왜 그 기술적 개입이 필요했는가"를 쓴다.
- Action은 본인의 기여가 드러나는 동사로 시작한다: 설계, 표준화, 자동화, 마이그레이션, 최적화, 검증, 문서화.
- Result는 Step 2의 measured outcome을 사용한다. 수치가 없으면 "정성 결과"로 표시하고 과장하지 않는다.
- 한 문장 bullet은 1~2줄로 제한한다. 긴 맥락은 "확장형"에 둔다.
- 여러 성과가 있으면 임팩트가 큰 순서로 최대 3개만 작성한다.
- "PAR만" 요청한 경우에도 Step 1~4의 최소 근거 추출을 내부적으로 먼저 수행한다. 이때 화면에는 Part A를 출력하지 않는다.
- **대표 PAR 포맷**: Problem / Action / Result을 각각 **레이블 상위 불릿 + 내용 들여쓰기 하위 불릿** 조합으로 쓴다. 레이블(`**Problem:**` 등)만 상위 불릿에 두고, 실제 내용은 그 아래 한 단계 들여쓴 하위 불릿에 둔다. 레이블과 내용을 같은 줄에 붙이지 않는다.
- **PAR 레이블 색상**: Notion 저장 시 상위 불릿의 레이블(`Problem:`/`Action:`/`Result:`)은 볼드 + 갈색(brown)으로 처리한다. 화면(터미널) 마크다운 출력에서는 색상 표현이 없으므로 볼드만 유지하고, Step 10 Notion append 및 notion-review 교정 단계에서 brown 색상을 적용한다.

```
### PAR 성과 문장

**대표 PAR**
- **Problem:**
    - {Situation + Pain/Risk/Goal: 어떤 맥락에서 왜 중요한 문제였는가}
- **Action:**
    - {내가 수행한 핵심 판단과 실행}
- **Result:**
    - {측정된 결과: 수치 우선, 없으면 검증된 정성 결과}

**이력서 bullet** (표준 포맷: `~/.claude/docs/resume-format-convention.md`)
- {Situation/원인}(으)로 {Before 수치가 있으면 포함} {현상}을/를, {핵심 Action: 구체적 방법·기술}(으)로
  {Result: 대괄호 또는 화살표 표기} {명사형 종결}.
  - 예: "CronJob 지연으로 인한 데이터 유실을 gp2 burst 한계 기반 디스크 병목으로 진단, gp3 전환으로
    유실 [N건→0건] / 지연 [N분→M분] 해소"
  - "~했습니다"/"~합니다" 같은 문장형 종결 금지, 반드시 명사형(해결/개선/절감/전환/구현 등)으로 끝낸다.

**성과평가용 확장형**
- {Situation과 Pain/Risk/Goal}. 이를 해결하기 위해 {판단 기준과 실행 내용}. 그 결과 {측정된 효과와 팀/운영 관점의 의미}.
```

**품질 체크:**
- Problem 없이 Action만 있으면 단순 업무 나열이다.
- Result 없이 Action만 있으면 성과가 아니라 활동 기록이다.
- Action 없이 Result만 있으면 본인의 기여가 불명확하다.
- 숫자가 없을 때는 "추적 가능성 개선", "롤백 경로 명확화"처럼 검증 가능한 변화만 쓴다.
- 이력서 bullet이 "~했습니다"로 끝나면 표준 포맷 위반이다. 명사형 종결로 다시 쓴다.

# Part B: 성장 회고 (Growth Retrospective)

## Step 5: 잘 한 것 (Keep)

대화에서 실제로 잘 수행된 것만 꼽는다. 빈 칭찬 금지.

**판단 기준:**
- 체계적 접근으로 시간을 단축한 부분
- 증거를 확보한 후 결론을 낸 부분
- 팀 원칙(생산성 > 비용 > 안정성)에 부합하는 의사결정
- 근본 원인까지 파고든 탐구

각 항목에 **왜 잘 한 것인지**를 반드시 붙인다.

```
### 잘 한 것

- **{항목}**: {왜 좋은 접근이었는지 1줄}
```

---

## Step 6: 더 고민했으면 좋았을 것 (Improve)

이번 작업에서 **놓쳤거나**, **더 깊이 팠어야 하거나**, **다른 방향으로 접근했으면 좋았을 것**.
"다음엔 이렇게 한다"는 구체적 대안까지 포함한다.

**작업 타입별 점검 차원:**

### RCA 타입 점검 항목
- [ ] known-issues.md를 먼저 확인했는가?
- [ ] 메트릭 → 로그 → 이벤트 순서로 레이어를 교차 확인했는가?
- [ ] 영향받은 컴포넌트가 1개라는 가정을 검증했는가?
- [ ] "왜 이 시각에 발생했는가"를 특정했는가?
- [ ] 어떤 스트림/파드/프로세스가 원인인지 최소 단위로 특정했는가?
- [ ] 실제 설정값을 직접 확인했는가, 아니면 default를 가정했는가?
- [ ] 수정 후 검증 기준을 정량적으로 명시했는가?
- [ ] 이 알럿이 더 빨리 탐지됐을 수 있는 파생 알럿 부재를 확인했는가?
- [ ] User-facing SLO 영향을 메트릭으로 검증했는가?

### Deployment 타입 점검 항목
- [ ] Prod 배포 정책(replica, PDB, affinity) 체크리스트를 완료했는가?
- [ ] Blast radius를 명시적으로 평가했는가?
- [ ] 롤백 경로와 소요 시간을 확인했는가?
- [ ] 배포 후 ArgoCD sync + pod 상태를 검증했는가?
- [ ] 이 변경이 다른 환경(dev/stg)에도 동일하게 적용돼야 하는가?

### Design 타입 점검 항목
- [ ] 대안을 최소 2개 이상 비교했는가?
- [ ] "6개월 후" 관점의 2차 효과를 고려했는가?
- [ ] 되돌리기 어려운 결정(one-way door)을 명시했는가?
- [ ] [work-definition-framework.md](~/workspace/riiid/kubernetes/devops-wiki/01-decisions/work-definition-framework.md)의 유형/레벨 판단과 Notion ROI가 일치하는가? (비용·운영 부담 포함)
- [ ] 이 설계가 야기할 기술 부채를 정량화했는가?

### Development 타입 점검 항목
- [ ] 근본 원인을 고쳤는가, 증상을 고쳤는가?
- [ ] 변경 영향 범위가 최소화됐는가?
- [ ] 테스트 또는 검증 단계가 있었는가?
- [ ] 기존 패턴과 일치하는가, 이탈 이유를 명시했는가?

**출력 포맷:**
```
### 더 고민했으면 좋았을 것

**{번호}. {놓친 것 제목}**
- 상황: {어떤 시점에 어떤 판단을 했는가}
- 문제: {왜 부족했는가}
- 대안: {어떻게 했으면 됐는가: 구체적 명령어/접근법 포함}
```

## Step 7: 알아야 할 개념 / 지식 갭 (Learn)

이번 작업에서 **몰라서 놓쳤거나**, **알았다면 더 빨리 해결했을** 개념과 지식.

각 항목을 세 레벨로 제시한다:
1. **개념**: 무엇을 알아야 하는가
2. **왜**: 이것을 알았으면 이번 작업에서 어떻게 달라졌는가
3. **어떻게**: 어떻게 학습할 수 있는가 (문서 링크 또는 실습 방법)

```
### 알아야 할 개념

**{개념명}**
- 이번 작업과의 연결: {알았다면 어떻게 달라졌는가}
- 학습 방법: {공식 문서 / 실습 / 관련 코드 경로}
```

## Step 8: 파생 액션 (Follow-up)

리뷰 결과에서 자연스럽게 도출되는 후속 작업. 사용자가 결정할 수 있도록 선택지로 제시.

**자동 분류 기준:**

| 신호 | 제안 액션 |
|------|-----------|
| Step 4의 "남은 것"(미달/잔여 목표) | 후속 작업 등록 제안 |
| 파생 알럿 부재 발견 | 알럿 추가 PR |
| 반복 가능한 함정 | `/devops:wiki:ingest guardrail` |
| 정상 이상현상 패턴 | `/devops:wiki:ingest known` |
| 배우면 좋을 개념 | `/learn {개념명}` |
| 미해결 조사 항목 | Notion Task 등록 제안 |
| 설정 개선 기회 | GitOps PR 제안 |

```
### 파생 액션 제안

**즉시**
- {액션}: {이번 작업에서 발견된 직접 gap}

**이번 주**
- {액션}: {재발 방지 또는 학습}

**언젠가**
- {액션}: {개선 기회이지만 긴급하지 않음}
```

해당 우선순위에 제안 액션이 없으면 그 섹션을 생략한다.

## Step 9: 한 줄 요약

회고 전체를 한 문장으로 압축한다.

> 형식: "{잘 한 핵심} + {가장 중요한 개선 포인트}"
> 예: "증거 기반으로 끝까지 파고드는 건 잘 됐지만, 실제 설정값 직접 확인과 파생 알럿 추가 기회를 놓쳤다."

---

## 출력 전체 포맷

출력 생성 전 `assets/output-template.md`를 Read하여 해당 포맷을 기준으로 작성한다.
단, 사용자가 "성과만", "PAR만", "회고만"처럼 범위를 좁히면 해당 파트만 출력한다.
"PAR만" 요청의 경우 Step 1~4는 내부 근거 추출로만 수행하고, 출력에는 Part A.5만 포함한다.

---

## 주의사항

**Part A (성과 측정)**
- **목표 식별 선행**: 최초 목표를 못 찾으면 측정 불가. 대화 첫 요청을 반드시 확인한다.
- **증거 없는 성과 금지**: 대화에서 확인된 산출물/메트릭만 기록. "아마 개선됐을 것" 금지.
- **포장 금지**: 부분 달성을 달성으로, 미달을 부분으로 올려치지 않는다.
- **계획 외 결과 누락 금지**: side effect(긍정·부정 모두)는 성과의 일부다.

**Part A.5 (PAR 성과 문장)**
- **근거 없는 포장 금지**: Step 2에서 측정되지 않은 효과를 Result에 넣지 않는다.
- **기술명 나열 금지**: Kubernetes, ArgoCD, Terraform 같은 기술명은 Action의 수단일 뿐이며 성과 자체가 아니다.
- **주어 누락 금지**: 팀 전체 성과와 본인 기여를 혼동하지 않는다. 본인이 한 판단과 실행을 명시한다.
- **출력 개수 제한**: 기본은 대표 PAR 1개. 성과가 여러 개면 최대 3개까지 임팩트 순으로 작성한다.
- **Situation 누락 금지**: Problem이 Pain/Risk만 담고 있으면 맥락이 약해진다. Situation을 1개 절로 압축한다.

**Part B (성장 회고)**
- **길이 경계**: Step 6 항목은 최대 5개, Step 7은 최대 4개. 그 이상은 핵심이 희석된다.
- **추측 금지**: 대화에 근거 없는 "아마 이것도 놓쳤을 것" 금지. 실제 확인된 것만.
- **비판 톤 금지**: "~를 안 했다"가 아니라 "~를 했으면 {구체적 효과}를 얻었을 것"으로 표현.
- **작업 타입 우선**: 점검 항목은 감지된 타입 것만 적용. 해당 없는 항목은 건너뛴다.
- **파생 액션은 제안만**: 사용자가 선택. 자동 실행하지 않는다.

## Step 10: Notion 저장 & 교정 파이프라인 (독립 실행 시)

**적용 범위**: `/task:review`를 독립 실행할 때만 수행한다. alfred gate 6단계에서 인라인 실행된 경우 alfred가 저장을 담당하므로 이 단계를 건너뛴다.

최종 리뷰 출력 전체를 기준으로 저장 여부를 1회만 제안한다. 사용자가 생략해도 재촉하지 않는다.

```
이 리뷰를 Notion Task 본문에 저장하시겠습니까?
저장 후 notion-review → notion-refactoring 파이프라인이 문서를 자동 교정합니다. (Y/N)
```

**Y 선택 시:**

1. **page_id 확정**: 대화 맥락의 가장 최근 활성 Task `page_id`를 사용한다. 없으면 사용자에게 Task명 확인 후 검색:
   ```bash
   python3 /Users/changhwan/.claude/skills/tasks:manage/scripts/notion-task.py search-tasks --status active
   ```
   Task명 fuzzy 매칭으로 `page_id`를 특정한다. 여러 후보가 있으면 1회 확인.

2. **리뷰 본문 append**:
   ```bash
   cat > /tmp/task-review-{slug}.md << 'EOF'
   ---
   ## task:review
   {최종 리뷰 출력 전체}
   EOF

   python3 /Users/changhwan/.claude/skills/tasks:manage/scripts/notion-task.py append-content \
     --page-id <page_id> \
     --content-file /tmp/task-review-{slug}.md
   ```
   - `{slug}` = Task명 소문자·하이픈 변환.
   - 마크다운 표는 미지원이므로 review 본문에 표가 있으면 bullet로 대체.
   - 저장 후: "리뷰가 Notion Task 본문에 저장됐습니다 (blocks_appended: {N})."
   - 실패 시: "저장 실패: `page_id`와 권한을 확인해 주십시오." 후 종료.

3. **notion-review → notion-refactoring 파이프라인 실행** (자동):
   저장 성공 시 즉시 실행한다. 두 에이전트는 background로 띄워 사용자와의 대화 흐름과 병렬로 진행하되, 에이전트 간에는 반드시 순차(파이프라인)로 실행한다.

   - `Agent(subagent_type="notion-review", prompt="{page_id}")` 로 페이지 교정 (background).
   - em dash(U+2014), 이모지, 문장 스타일 위반을 자동 감지·수정.
   - 완료 알림 도착 시 교정 결과 1줄 보고:
     ```
     → notion-review: {N}건 교정 완료.
     ```
   - 교정이 없으면: "→ notion-review: 교정 불필요 (문서 스타일 적합)."
   - 실패 시: "→ notion-review 실패: Notion 페이지에서 수동 검토를 권합니다." 후 종료 (refactoring 체이닝 생략).
   - **notion-refactoring 체이닝 (자동)**: notion-review 리포트의 `### Refactor handoff` 블록에 `findings`가 1건 이상이면, 블록 원문을 그대로 prompt로 전달해 background로 실행한다:
     `Agent(subagent_type="notion-refactoring", prompt="{Refactor handoff 블록 원문}")`
     - 리뷰가 보고만 한 주관적 지적(문장 분리·병합, 군더더기, 코드 블록 오용, 구조·톤, 이력서 bullet 포맷)을 의미 보존 범위에서 반영한다.
     - 완료 알림 도착 시 1줄 보고: "→ notion-refactoring: {M}건 반영, {K}건 보류."
     - `findings: 0`이면 띄우지 않는다.
   - **동시 실행 금지 (하드룰)**: 두 에이전트는 같은 페이지의 전체 markdown을 교체하므로 절대 동시에 띄우지 않는다 (동시 쓰기는 한쪽 수정이 유실됨). review 완료 확인 → refactoring 순서를 지킨다.

**N 선택 시:** 화면 출력으로만 종료. "필요하시면 언제든 `/alfred gate`에서 저장하실 수 있습니다."

# task:review — 작업 후 성과 측정 & 성장 회고 스킬

작업 완료 후 대화 맥락 전체를 분석하여 두 가지를 수행한다.

- **Part A — 성과 측정**: 최초 목표 대비 무엇을 달성했는가를 증거 기반으로 측정한다. (사실 확정)
- **Part B — 성장 회고**: 다음 번에 더 잘 하기 위한 구체적 인사이트를 생성한다. (성찰)

**순서가 중요하다.** 성과 측정(사실)을 먼저 확정한 뒤 성장 회고(성찰)로 넘어간다.
"무엇을 달성했는가"가 확정돼야 "어떻게 더 잘 할 수 있었는가"가 의미를 가진다.

**실행 범위:** 기본은 Part A + Part B 모두 수행한다.
사용자가 "성과만" 요청하면 Part A만, "회고만" 요청하면 Part B만 수행한다.

---

## 핵심 원칙

**공통**
- **증거 기반**: 대화에서 실제 발생한 일만 기반으로 분석. 추측 금지.
- **작업 타입 인식**: RCA / 배포 / 설계 / 개발 / 트러블슈팅별 차원이 다르다.

**Part A (성과 측정) 전용**
- **목표 baseline 우선**: 성과는 항상 "최초 목표 대비"로 측정한다. 목표 식별이 선행돼야 한다.
- **정량 우선**: before→after 수치를 우선한다. 불가능할 때만 정성으로.
- **산출물과 효과 분리**: PR/commit(산출물)과 알럿 해소/비용 절감(효과)을 구분한다. 효과가 진짜 성과다.
- **정직한 판정**: 미달성을 달성으로 포장하지 않는다. 부분 달성은 부분으로, 미달은 미달로.

**Part B (성장 회고) 전용**
- **비판보다 성장**: 무엇이 나빴는지가 아니라 무엇을 달리 했으면 결과가 달라졌는지.
- **구체성**: "더 잘 확인해야 했다" 금지. "어떤 명령어로 무엇을 확인했어야 했는지" 명시.
- **학습 연결**: 놓친 것은 반드시 "왜 몰랐는지" 또는 "어떤 개념을 알면 됐는지"로 연결.

## Step 0 — 작업 타입 감지

대화 맥락에서 이번 작업의 주 타입을 결정한다. 복수 타입이면 주 타입 1개 + 부 타입 병기.

| 타입 | 감지 신호 |
|------|-----------|
| **RCA** | 알럿 URL 분석, kubectl 조사, 메트릭 쿼리, 5 Whys, root cause 탐색 |
| **Deployment** | image tag 업데이트, PR 생성, ArgoCD sync, canary, rollout |
| **Design** | Tech Spec, 아키텍처 논의, 트레이드오프 비교, ADR |
| **Development** | 코드 수정, 버그 수정, 기능 추가, 리팩터링 |
| **Troubleshooting** | 특정 에러 해결, 설정 수정, 디버깅 (RCA보다 범위 작음) |
| **Operations** | 인프라 변경, 업그레이드, 정리 작업, 모니터링 설정 |

# Part A — 성과 측정 (Outcome Measurement)

## Step 1 — 작업 분석 & 최초 목표 식별

대화를 처음부터 끝까지 스캔하여 "무엇을 하려 했고 무엇을 했는가"를 확정한다.
성과 측정의 baseline은 **최초 목표**이므로 이 단계가 가장 중요하다.

**식별 항목:**
- **최초 목표(명시적)**: 사용자가 대화 시작 시 요청한 것. 원문 의도 그대로.
- **실질 목표(암묵적)**: 작업 중 드러난 진짜 목표. 사용자가 X를 요청했어도 근본 문제가 Y였다면 Y를 명시.
- **수행한 작업**: 실제로 실행된 핵심 작업 (1-3줄).
- **범위 변화**: 작업 중 목표가 추가/축소/전환됐는가 (scope creep / pivot). 없으면 "없음".

```
**작업 타입**: {주 타입} (+ {부 타입})
**최초 목표(명시적)**: {사용자 최초 요청}
**실질 목표(암묵적)**: {작업 중 드러난 진짜 목표 — 명시적과 같으면 "동일"}
**수행한 작업**: {실제로 한 것 1-3줄}
**범위 변화**: {추가/축소/전환된 것 — 없으면 "없음"}
```

---

## Step 2 — 성과 측정

실제로 산출/달성된 것을 측정한다. **주장 금지, 증거 필수** — 대화에서 확인된 산출물/메트릭만.

**측정 차원** (팀 가중치 순: 생산성 > 비용 > 안정성):

| 차원 | 측정 대상 예시 |
|------|---------------|
| **생산성** | 자동화, 병목 해소, 작업 시간 단축, self-service 가능해진 것 |
| **비용** | 리소스/네트워크 비용 절감 (정량 금액 또는 용량) |
| **안정성** | 알럿 해소, 장애 복구, 가용성/SLO 개선 |
| **지식/부채** | 문서화, 런북, 기술 부채 해소 (또는 의도적으로 생성한 부채) |

**산출물 vs 효과를 분리한다:**
- **산출물(Artifacts)**: PR/commit/문서/설정 변경 — 무엇을 만들었는가
- **효과(Outcomes)**: 알럿 해소, 88%→41%, 145GB 회수 — 무엇이 실제로 달라졌는가

각 효과는 정량(before→after)을 우선하고, 정량 불가 시 정성으로 명시한다.

```
**산출물**
- {PR/commit/문서 등 — 식별자 포함}

**측정된 효과**

| 차원 | 효과 | 측정값 (before→after) | 증거 |
|------|------|----------------------|------|
| {생산성/비용/안정성/지식} | {달라진 것} | {수치 또는 정성} | {대화 근거} |
```

---

## Step 3 — 목표 달성 여부 판정

Step 1의 목표와 Step 2의 성과를 대조하여 판정한다.

**판정 등급:**

| 등급 | 의미 |
|------|------|
| **달성** | 목표 100% 충족 |
| **부분 달성** | 핵심은 충족, 일부 미완 (미완 항목 + 이유 명시) |
| **미달성** | 목표 미충족 (이유 + blocker 명시) |
| **초과 달성** | 목표 외 추가 가치까지 산출 |

규칙:
- 목표가 복수면 **항목별로** 판정한다.
- 측정 기준은 **실질 목표(암묵적)** 우선. 명시적 목표만 달성하고 근본 문제를 못 풀었다면 부분 달성이다.
- **계획 외 결과(side effect)**를 반드시 별도 기록한다 — 긍정(추가 발견·개선)과 부정(새로 생긴 부채·리스크) 모두.

```
| 목표 항목 | 판정 | 근거 | 미달/잔여 |
|----------|------|------|----------|
| {목표1} | 달성 | {성과 증거} | - |
| {목표2} | 부분 달성 | {부분 성과} | {남은 것 + 이유} |

**계획 외 결과**
- 긍정: {side benefit — 없으면 "없음"}
- 부정: {새로 생긴 리스크/부채 — 없으면 "없음"}
```

## Step 4 — 성과 요약

Part A 전체를 압축한다.

```
**전체 달성도**: {종합 판정 — 예: 핵심 목표 달성, 후속 1건 잔여}
**핵심 성과**: {가장 임팩트 큰 성과 1줄 — 정량값 포함}
**남은 것**: {미달 항목 또는 후속 작업 — 없으면 "없음"}
```

# Part B — 성장 회고 (Growth Retrospective)

## Step 5 — 잘 한 것 (Keep)

대화에서 실제로 잘 수행된 것만 꼽는다. 빈 칭찬 금지.

**판단 기준:**
- 체계적 접근으로 시간을 단축한 부분
- 증거를 확보한 후 결론을 낸 부분
- 팀 원칙(생산성 > 비용 > 안정성)에 부합하는 의사결정
- 근본 원인까지 파고든 탐구

각 항목에 **왜 잘 한 것인지**를 반드시 붙인다.

```
### 잘 한 것

- **{항목}**: {왜 좋은 접근이었는지 1줄}
```

---

## Step 6 — 더 고민했으면 좋았을 것 (Improve)

이번 작업에서 **놓쳤거나**, **더 깊이 팠어야 하거나**, **다른 방향으로 접근했으면 좋았을 것**.
"다음엔 이렇게 한다"는 구체적 대안까지 포함한다.

**작업 타입별 점검 차원:**

### RCA 타입 점검 항목
- [ ] known-issues.md를 먼저 확인했는가?
- [ ] 메트릭 → 로그 → 이벤트 순서로 레이어를 교차 확인했는가?
- [ ] 영향받은 컴포넌트가 1개라는 가정을 검증했는가?
- [ ] "왜 이 시각에 발생했는가"를 특정했는가?
- [ ] 어떤 스트림/파드/프로세스가 원인인지 최소 단위로 특정했는가?
- [ ] 실제 설정값을 직접 확인했는가, 아니면 default를 가정했는가?
- [ ] 수정 후 검증 기준을 정량적으로 명시했는가?
- [ ] 이 알럿이 더 빨리 탐지됐을 수 있는 파생 알럿 부재를 확인했는가?
- [ ] User-facing SLO 영향을 메트릭으로 검증했는가?

### Deployment 타입 점검 항목
- [ ] Prod 배포 정책(replica, PDB, affinity) 체크리스트를 완료했는가?
- [ ] Blast radius를 명시적으로 평가했는가?
- [ ] 롤백 경로와 소요 시간을 확인했는가?
- [ ] 배포 후 ArgoCD sync + pod 상태를 검증했는가?
- [ ] 이 변경이 다른 환경(dev/stg)에도 동일하게 적용돼야 하는가?

### Design 타입 점검 항목
- [ ] 대안을 최소 2개 이상 비교했는가?
- [ ] "6개월 후" 관점의 2차 효과를 고려했는가?
- [ ] 되돌리기 어려운 결정(one-way door)을 명시했는가?
- [ ] 팀 원칙(ROI, 비용, 운영 부담)과의 정합성을 확인했는가?
- [ ] 이 설계가 야기할 기술 부채를 정량화했는가?

### Development 타입 점검 항목
- [ ] 근본 원인을 고쳤는가, 증상을 고쳤는가?
- [ ] 변경 영향 범위가 최소화됐는가?
- [ ] 테스트 또는 검증 단계가 있었는가?
- [ ] 기존 패턴과 일치하는가, 이탈 이유를 명시했는가?

**출력 포맷:**
```
### 더 고민했으면 좋았을 것

**{번호}. {놓친 것 제목}**
- 상황: {어떤 시점에 어떤 판단을 했는가}
- 문제: {왜 부족했는가}
- 대안: {어떻게 했으면 됐는가 — 구체적 명령어/접근법 포함}
```

## Step 7 — 알아야 할 개념 / 지식 갭 (Learn)

이번 작업에서 **몰라서 놓쳤거나**, **알았다면 더 빨리 해결했을** 개념과 지식.

각 항목을 세 레벨로 제시한다:
1. **개념**: 무엇을 알아야 하는가
2. **왜**: 이것을 알았으면 이번 작업에서 어떻게 달라졌는가
3. **어떻게**: 어떻게 학습할 수 있는가 (문서 링크 또는 실습 방법)

```
### 알아야 할 개념

**{개념명}**
- 이번 작업과의 연결: {알았다면 어떻게 달라졌는가}
- 학습 방법: {공식 문서 / 실습 / 관련 코드 경로}
```

## Step 8 — 파생 액션 (Follow-up)

리뷰 결과에서 자연스럽게 도출되는 후속 작업. 사용자가 결정할 수 있도록 선택지로 제시.

**자동 분류 기준:**

| 신호 | 제안 액션 |
|------|-----------|
| Step 4의 "남은 것"(미달/잔여 목표) | 후속 작업 등록 제안 |
| 파생 알럿 부재 발견 | 알럿 추가 PR |
| 반복 가능한 함정 | `/devops:wiki:ingest guardrail` |
| 정상 이상현상 패턴 | `/devops:wiki:ingest known` |
| 배우면 좋을 개념 | `/learn {개념명}` |
| 미해결 조사 항목 | Notion Task 등록 제안 |
| 설정 개선 기회 | GitOps PR 제안 |

```
### 파생 액션 제안

| 우선순위 | 액션 | 이유 |
|----------|------|------|
| 즉시 | {액션} | {이번 작업에서 발견된 직접 gap} |
| 이번 주 | {액션} | {재발 방지 또는 학습} |
| 언젠가 | {액션} | {개선 기회이지만 긴급하지 않음} |
```

## Step 9 — 한 줄 요약

회고 전체를 한 문장으로 압축한다.

> 형식: "{잘 한 핵심} + {가장 중요한 개선 포인트}"
> 예: "증거 기반으로 끝까지 파고드는 건 잘 됐지만, 실제 설정값 직접 확인과 파생 알럿 추가 기회를 놓쳤다."

## 주의사항

**Part A (성과 측정)**
- **목표 식별 선행**: 최초 목표를 못 찾으면 측정 불가. 대화 첫 요청을 반드시 확인한다.
- **증거 없는 성과 금지**: 대화에서 확인된 산출물/메트릭만 기록. "아마 개선됐을 것" 금지.
- **포장 금지**: 부분 달성을 달성으로, 미달을 부분으로 올려치지 않는다.
- **계획 외 결과 누락 금지**: side effect(긍정·부정 모두)는 성과의 일부다.

**Part B (성장 회고)**
- **길이 경계**: Step 6 항목은 최대 5개, Step 7은 최대 4개. 그 이상은 핵심이 희석된다.
- **추측 금지**: 대화에 근거 없는 "아마 이것도 놓쳤을 것" 금지. 실제 확인된 것만.
- **비판 톤 금지**: "~를 안 했다"가 아니라 "~를 했으면 {구체적 효과}를 얻었을 것"으로 표현.
- **작업 타입 우선**: 점검 항목은 감지된 타입 것만 적용. 해당 없는 항목은 건너뛴다.
- **파생 액션은 제안만**: 사용자가 선택. 자동 실행하지 않는다.

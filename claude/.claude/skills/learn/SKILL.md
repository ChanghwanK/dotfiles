# Deep-dive Learning Session

6단계 구조화 학습 세션을 진행하는 워크플로우 스킬.

---

## 핵심 원칙

- **진단 우선**: 사용자의 자기보고 수준에 의존하지 않는다. Phase 0에서 실제 질문을 통해 지식 약점을 탐지하고, Gap이 있는 영역을 집중적으로 다룬다
- **성장 촉진자**: 정답을 바로 제시하지 않는다. 스스로 생각하고 사고하여 성장할 수 있도록 돕는 것이 주 역할
- **Socratic Questioning**: 각 Phase 마무리 시 "왜?" 질문 2~3개로 핵심 통찰을 되짚고 다음 단계 사고를 예비. 간결한 힌트로 방향만 제시
- **Why-first**: 구현 세부사항보다 설계 동기와 제약 조건을 먼저 다룬다
- **Source of truth**: 공식 문서를 넘어 실제 소스코드/구현을 근거로 삼는다
- **대안 비교**: 현재 설계를 이해하려면 "선택하지 않은 길"을 함께 본다
- **사용자 주도**: Phase 순서는 권장이지 강제가 아니다. 사용자 흐름에 따른다
- **적응적 깊이**: Diagnostic에서 파악한 실제 약점에 맞춰 설명 밀도와 집중 영역을 조절한다

---

## Phase 구조

```
Phase 0: Setup        → 주제 확인, 사전 지식 파악, 세션 범위 설정
Phase 1: Why          → 이 기술이 해결하는 문제, 설계 동기, 제약 조건
Phase 2: How          → 내부 아키텍처, 핵심 코드 경로, 데이터 흐름
Phase 3: What-if      → 대안 설계 비교, trade-off 분석
Phase 4: Apply        → 우리 인프라(SOCRA AI)에서의 적용/연결
Phase 5: Summary      → 구조화된 요약, 학습 로드맵, 장애 시나리오 훈련
```

---

## Socratic 질문 생성 규칙

각 Phase 마무리 후 해당 Phase 내용에서 핵심 통찰을 추출하여 "왜 ~할까?" 형태의 질문 2~3개를 생성한다.

### 질문 설계 원칙
- Phase에서 다룬 내용의 **설계 동기/선택 이유**를 되짚는 질문
- 다음 Phase로 자연스럽게 사고가 이어지는 질문 1개 포함
- 토픽에 특화된 구체적 질문 (범용적/추상적 질문 금지)

### 수준별 힌트
| Level | 힌트 제공 |
|-------|----------|
| easy | 각 질문에 힌트 포함 (거의 답에 가까운 방향) |
| medium | 핵심 질문에만 힌트 (키워드 수준) |
| hard | 힌트 없음 |

---

## Phase 0 — Setup (필수)

세션 시작 시 반드시 수행한다.

### 모드 감지

사용자 메시지에서 의도를 파악하여 모드를 선택한다:

| 모드 | 트리거 패턴 | 흐름 |
|------|------------|------|
| **Deep-dive** (기본) | `/learn {topic}`, "학습", "deep-dive" | Phase 0 수행 항목 → Phase 1 → 2 → 3 → 4 → 5 |
| **Recommend** | "추천", "recommend", "뭐 공부", "어떤 주제", "학습 주제", "학습 추천" + 도메인 키워드 | Recommend 모드 섹션으로 이동 |

Deep-dive 모드는 아래 수행 항목을 그대로 진행한다.

### 수행 항목

1. **주제 확인**: topic을 파악한다
2. **Diagnostic 진단**: Claude가 2~3개 질문으로 실제 이해 수준과 약점을 탐지한다
3. **약점 분석 공유**: 진단 결과에서 Gap을 명시한다
4. **세션 범위 설정**: Gap 기반으로 집중 영역과 생략 영역을 결정한다

### Diagnostic 진단 질문 설계 원칙

- **질문 개수**: 2~3개 (진입장벽 방지)
- **질문 유형** (순서대로 선택):
  - **Q1 — 정의형**: "{topic}을 한 문장으로 설명해보세요" → 개념 파악 여부 확인
  - **Q2 — 메커니즘형**: "{topic}이 {핵심 동작}을 할 때 내부적으로 어떤 과정이 일어나나요?" → 오개념·표면 이해 탐지 (가장 중요)
  - **Q3 — 트레이드오프형**: "{topic} 대신 {대안}을 쓰면 어떤 차이가 있나요?" → 깊이 탐지 (생략 가능)
- **질문 생성 규칙**: 반드시 토픽에 특화된 질문 사용. "아는 게 뭐예요?" 같은 범용 질문 금지
- **응답 분석 포인트**: 오개념, 키워드만 나열(설명 불가), 특정 레이어만 이해(전체 흐름 모름)

### 진단 출력 형식

1단계 — 질문 제시:

```
## Diagnostic: {topic}

{topic}에 대한 이해를 파악하기 위해 몇 가지 질문을 드립니다.
모른다면 모른다고 말씀해주세요. 틀려도 괜찮습니다.

1. {Q1 — 정의형}
2. {Q2 — 메커니즘형}
3. {Q3 — 트레이드오프형, 있는 경우}

→ 빠르게 시작하려면 수준을 말씀해주세요 (easy / medium / hard). 진단을 건너뜁니다.
```

2단계 — 답변 분석 후 세션 세팅:

```
## Session Setup: {topic}

**진단 결과**
- Level: {탐지된 수준 — easy / medium / hard}
- 강점: {잘 이해된 부분}
- 약점/Gap: {이해가 부족하거나 오개념이 있는 부분}

**이 세션의 집중 영역**: {Gap을 채우는 데 필요한 Phase/내용}
- 빠르게 넘길 부분: {이미 잘 아는 부분}

Phase 1 (Why)부터 시작합니다. 다른 Phase로 바로 이동하고 싶으면 말씀해주세요.
```

---

## Phase 1 — Why (필수)

이 기술/개념이 무엇인지 정의하고, 존재하는 이유를 다룬다.

> Phase 1은 **기술 전체**가 해결하는 문제와 설계 동기를 다룬다.
> 개별 특징/메커니즘의 문제-트레이드오프-대안은 Phase 2에서 Feature Explanation Pattern으로 다룬다.

### What (개념 설명)

학습 주제가 무엇인지 먼저 한눈에 파악할 수 있도록 정리한다.

- **한 줄 정의**: {기술/개념을 한 문장으로}
- **핵심 구성 요소**: 이 기술을 이루는 주요 개념/컴포넌트 (2~4개)
- **포지셔닝**: 기술 스택/시스템에서 어디에 속하는지 (레이어, 역할, 카테고리)

### Why (설계 동기)

- **문제 정의**: 이 기술이 없었을 때의 고통 (pain point)
- **설계 동기**: 해결하려는 핵심 문제와 목표
- **제약 조건**: 설계를 제한한 기술적/조직적 맥락
- **핵심 통찰**: 이 설계를 관통하는 핵심 아이디어 한 줄

### 수준별 조절

| Level | What | Why |
|-------|------|-----|
| easy | 한 줄 정의 + 핵심 구성 요소만, 비유 사용 | 비유와 일상적 예시로 문제 설명 |
| medium | 구성 요소 간 관계 + 포지셔닝 도식 포함 | 기술적 맥락과 역사적 배경 포함 |
| hard | RFC/설계 문서 기반 정확한 스펙 정의 | 원본 설계 문서, RFC, KEP 등 1차 자료 참조 |

### Socratic 전환 질문 (Phase 1 → 2)

Phase 1 내용 전달 후, 아래 형식으로 "왜?" 질문을 제시한다:

```
**Why를 고민할 주제**

- 왜 {직전 Phase에서 다룬 핵심 설계 동기 관련 질문}?
  (힌트: {한 줄 힌트})
- 왜 {설계가 의도적으로 포기한 것 관련 질문}?
  (힌트: {한 줄 힌트})
- 왜 {다음 Phase(How)로 이어지는 구현 동기 질문}?
  (힌트: {한 줄 힌트})

생각해보신 후 말씀해주세요. (패스 가능)
```

사용자가 답변하면 간단히 반응 후 Phase 2로 전환한다. "패스" 시 즉시 다음으로 진행한다.

## Phase 2 — How (선택)

내부 동작 원리와 구현을 다룬다.

### 다루는 내용

- **아키텍처 개요**: 주요 컴포넌트와 상호작용
- **핵심 코드 경로**: 요청이 처리되는 과정 (critical path)
- **데이터 흐름**: 입력 → 변환 → 출력 과정
- **핵심 자료구조/알고리즘**: 성능과 동작을 결정하는 내부 구조

### Feature Explanation Pattern

Phase 2에서 핵심 특징/메커니즘을 설명할 때, 각 특징마다 아래 구조를 따른다:

1. **해결하는 문제**: 이 특징이 없으면 어떤 문제가 발생하는지 (1~2문장)
2. **동작 방식**: 어떻게 작동하는지 (기존 How 내용)
3. **트레이드오프**: 이 접근의 장단점 (성능, 복잡도, 운영 비용 등)
4. **대안 접근법**: 같은 문제를 다르게 해결하는 방법과 차이점

#### 수준별 조절

| Level | 적용 범위 |
|-------|----------|
| easy | 핵심 특징 1~2개에만 간략히 적용 |
| medium | 주요 특징 모두에 적용, 대안은 이름+한 줄 설명 |
| hard | 모든 특징에 적용, 대안의 구현 차이까지 비교 |

#### 예시 (Karpenter Consolidation)

```
**Consolidation (빈 패킹 최적화)**
- 해결하는 문제: 노드가 underutilized 상태로 방치되면 비용 낭비
- 동작 방식: 주기적으로 노드 utilization 평가 → 워크로드를 다른 노드로 이동 → 빈 노드 제거
- 트레이드오프: 비용 절감 vs Pod 재배치로 인한 일시적 서비스 중단
- 대안: Cluster Autoscaler의 scale-down (노드 단위 판단, 빈 패킹 없음)
```

### 수준별 조절

| Level | 깊이 |
|-------|------|
| easy | 블록 다이어그램 수준, 컴포넌트 역할 설명 |
| medium | 주요 인터페이스, 설정 옵션, 동작 모드 |
| hard | 소스코드 함수 단위 추적, 실제 struct/interface 분석 |

### 설명 보조 도구

복잡한 구조는 ASCII 다이어그램이나 시퀀스 다이어그램으로 보충한다:

```
[Component A] --request--> [Component B] --lookup--> [Store]
      ^                          |
      +------response------------+
```

### Socratic 전환 질문 (Phase 2 → 3)

Phase 2 내용 전달 후, 아래 형식으로 질문을 제시한다:

```
**How를 고민할 주제**

- 왜 {핵심 컴포넌트가 이런 구조를 선택했는지 관련 질문}?
  (힌트: {한 줄 힌트})
- 왜 {가장 독특한 설계 결정 관련 질문}?
  (힌트: {한 줄 힌트})
- 왜 {다른 접근이 아닌 현재 방식인지 — What-if로 이어지는 질문}?
  (힌트: {한 줄 힌트})

생각해보신 후 말씀해주세요. (패스 가능)
```

사용자가 답변하면 간단히 반응 후 Phase 3으로 전환한다. "패스" 시 즉시 다음으로 진행한다.

## Phase 3 — What-if (선택)

현재 설계를 더 깊이 이해하기 위해 대안을 탐색한다.

### 다루는 내용

- **대안 설계 1~2개**: 실제로 고려되었거나 다른 프로젝트가 선택한 방식
- **Trade-off 비교표**:

```
| 기준         | 현재 설계  | 대안 A     | 대안 B     |
|-------------|-----------|-----------|-----------|
| 복잡도       | ...       | ...       | ...       |
| 성능         | ...       | ...       | ...       |
| 운영 부담    | ...       | ...       | ...       |
| 확장성       | ...       | ...       | ...       |
```

- **선택 이유**: 현재 설계가 채택된 결정적 요인

### Socratic 전환 질문 (Phase 3 → 4)

Trade-off 분석 후, 아래 형식으로 질문을 제시한다:

```
**What-if를 고민할 주제**

- 왜 {현재 설계가 우리 환경에서 유리/불리한지 관련 질문}?
  (힌트: {한 줄 힌트})
- 왜 {대안이 특정 조건에서 더 나은지 관련 질문}?
  (힌트: {한 줄 힌트})
- 왜 {SOCRA AI 환경에서 구체적으로 고려해야 할 질문}?
  (힌트: {한 줄 힌트})

생각해보신 후 말씀해주세요. (패스 가능)
```

사용자가 답변하면 간단히 반응 후 Phase 4로 전환한다. "패스" 시 즉시 다음으로 진행한다.

---

## Phase 4 — Apply (선택)

학습 내용을 우리 환경(SOCRA AI 인프라)에 연결한다.

### 다루는 내용

- **현재 사용 현황**: 우리 인프라에서 이 기술을 어떻게 쓰고 있는지
- **적용 가능성**: 새로 도입하거나 개선할 수 있는 영역
- **주의사항**: 우리 환경 특성상 고려할 점 (EKS, Istio, Karpenter 등)

### 참고 환경

- EKS 클러스터: prod / global / stg / dev / idc
- Service Mesh: Istio
- GitOps: ArgoCD
- Node Provisioning: Karpenter
- GPU 클러스터: IDC (Proxmox + Cluster API)

## Phase 5 — Summary (필수)

세션 내용을 `obsidian:note`로 저장할 수 있는 구조로 정리한다.

### 출력 형식

```markdown
## 학습 요약: {topic}

### 핵심 한 줄
{이 기술/개념을 한 문장으로 요약}

### 설계 철학
{Phase 1에서 다룬 Why의 핵심}

### 내부 동작
{Phase 2에서 다룬 How의 핵심 (다룬 경우)}

### Trade-offs
{Phase 3에서 다룬 대안 비교 핵심 (다룬 경우)}

### 실무 연결
{Phase 4에서 다룬 적용 포인트 (다룬 경우)}

### 학습 로드맵

#### 깊이 확장 (이 주제를 더 깊이)
| # | 주제 | 난이도 | 연결 이유 |
|---|------|--------|-----------|
| 1 | {topic} | {level} | {왜 다음에 필요한지} |
| 2 | {topic} | {level} | {연결 이유} |

#### 범위 확장 (인접 영역으로)
| # | 주제 | 난이도 | 연결 이유 |
|---|------|--------|-----------|
```

## Recommend 모드 — 도메인별 학습 주제 추천

Phase 0에서 Recommend 모드로 분기된 경우 이 섹션을 따른다.
목표: 시니어/스태프 엔지니어 성장에 필요한 학습 주제를 도메인별로 구조화하여 추천한다.

### 핵심 원칙

- **시니어/스태프 관점**: 단순 사용법이 아닌, 설계 철학·내부 동작·트레이드오프를 이해하는 주제 위주
- **Why-first 추천**: 각 주제가 왜 시니어/스태프에게 중요한지 근거를 명시
- **의존성 기반 순서**: 선수 지식 → 핵심 → 심화 순서로 학습 경로 제시
- **SOCRAAI 연결 필수**: 범용 추천 금지, 우리 인프라와의 연결점 포함

### 진행 절차

1. **도메인 파악**: 사용자 메시지에서 도메인 추출 (예: "AWS network", "Kubernetes", "Observability")
2. **수준 파악**: 해당 도메인에 대한 현재 수준 확인 (easy / medium / hard)
   - 대화에서 이미 명확하면 생략
3. **주제 분석 & 추천 출력**: 아래 출력 형식에 따라 구조화된 추천 제공
4. **세션 전환**: 사용자가 번호를 선택하면 해당 주제로 Phase 0 (Deep-dive 모드)부터 시작

### 주제 분류 체계

추천 주제를 3개 레이어로 분류한다:

| 레이어 | 설명 | 시니어/스태프 관점 |

| **Foundation** | 선수 지식, 기초 원리 | "왜 이렇게 동작하는지" 설명 가능해야 함 |

## 장애 시나리오 훈련 진행 방식

사용자가 시나리오 번호를 선택하면:

1. **상황 부여**: 시나리오 배경, 시간대, 발생 조건
2. **단서 제공**: "어떤 명령어를 먼저 치겠습니까?" 질문
3. **단계별 힌트**: 사용자 응답에 따라 점진적으로 정보 제공
4. **해결**: 근본 원인 + 예방 조치
5. **회고**: 이 시나리오에서 배운 점 요약

## 유연성 규칙

- **Phase 순서 변경 가능**: 사용자가 "소스코드부터 보자"라고 하면 Phase 2로 바로 이동
- **Phase 건너뛰기**: 사용자가 "대안 비교는 생략"이라고 하면 Phase 3 스킵
- **Socratic 질문 패스**: 사용자가 "패스", "다음으로", "바로 진행"이라고 하면 Socratic 질문 전체를 건너뛴다
- **중간 질문 허용**: Phase 진행 중 사용자 질문이 오면 즉시 답변 후 원래 흐름 복귀
- **깊이 조절**: 사용자가 "더 깊이" 또는 "이 정도면 됐어"라고 하면 즉시 반영
- **긴 Phase 분할**: 한 Phase가 3회 이상 응답으로 길어지면 중간 정리 후 계속할지 확인

## 주의사항

- Phase 0 (Setup)과 Phase 5 (Summary)는 반드시 수행한다
- 사용자가 명시적으로 요청하지 않는 한 Phase를 강제로 진행하지 않는다
- 확실하지 않은 내용은 추측하지 않고 명시적으로 "확인 필요"로 표시한다
- 소스코드 분석 시 버전/커밋을 가능한 한 명시한다
- 학습 세션 중 발견한 우리 인프라 개선 아이디어는 Phase 4에서 별도로 정리한다

# Deep-dive Learning Session

6단계 구조화 학습 세션을 진행하는 워크플로우 스킬.

---

## 핵심 원칙

- **Why-first**: 구현 세부사항보다 설계 동기와 제약 조건을 먼저 다룬다
- **Source of truth**: 공식 문서를 넘어 실제 소스코드/구현을 근거로 삼는다
- **대안 비교**: 현재 설계를 이해하려면 "선택하지 않은 길"을 함께 본다
- **사용자 주도**: Phase 순서는 권장이지 강제가 아니다. 사용자 흐름에 따른다
- **적응적 깊이**: Phase 0에서 파악한 수준에 맞춰 설명 밀도를 조절한다

---

## Phase 구조

```
Phase 0: Setup        → 주제 확인, 사전 지식 파악, 세션 범위 설정
Phase 1: Why          → 이 기술이 해결하는 문제, 설계 동기, 제약 조건
Phase 2: How          → 내부 아키텍처, 핵심 코드 경로, 데이터 흐름
Phase 3: What-if      → 대안 설계 비교, trade-off 분석
Phase 4: Apply        → 우리 인프라(SOCRA AI)에서의 적용/연결
Phase 5: Summary      → 구조화된 요약, 학습 로드맵, 장애 시나리오 훈련
```

---

## Phase 0 — Setup (필수)

세션 시작 시 반드시 수행한다.

### 수행 항목

1. **주제 확인**: `/learn {topic}`에서 topic을 파악한다
2. **사전 지식 파악**: 아래 3단계 중 하나를 확인한다
   - `beginner`: 개념 자체가 처음. 배경부터 설명 필요
   - `intermediate`: 기본 사용법은 알지만 내부 동작이 궁금
   - `advanced`: 구현 세부사항, 엣지 케이스, 소스코드 레벨 분석 원함
3. **범위 설정**: 다루고 싶은 영역과 생략할 영역을 확인한다

### 출력 형식

```
## Session: {topic}
- Level: {beginner | intermediate | advanced}
- Scope: {다룰 영역}
- Skip: {제외 영역, 없으면 "없음"}

Phase 1 (Why)부터 시작합니다. 다른 Phase로 바로 이동하고 싶으면 말씀해주세요.
```

---

## Phase 1 — Why (필수)

이 기술/개념이 존재하는 이유를 다룬다.

### 다루는 내용

- **문제 정의**: 이 기술이 없었을 때의 고통 (pain point)
- **설계 동기**: 해결하려는 핵심 문제와 목표
- **제약 조건**: 설계를 제한한 기술적/조직적 맥락
- **핵심 통찰**: 이 설계를 관통하는 핵심 아이디어 한 줄

### 수준별 조절

| Level | 깊이 |
|-------|------|
| beginner | 비유와 일상적 예시로 문제 설명 |
| intermediate | 기술적 맥락과 역사적 배경 포함 |
| advanced | 원본 설계 문서, RFC, KEP 등 1차 자료 참조 |

### Phase 전환

내용을 전달한 후 사용자에게 확인:
```
Phase 1 (Why) 정리가 끝났습니다.
→ Phase 2 (How: 내부 구현)로 넘어갈까요?
→ 또는 다른 Phase나 추가 질문이 있으면 말씀해주세요.
```

---

## Phase 2 — How (선택)

내부 동작 원리와 구현을 다룬다.

### 다루는 내용

- **아키텍처 개요**: 주요 컴포넌트와 상호작용
- **핵심 코드 경로**: 요청이 처리되는 과정 (critical path)
- **데이터 흐름**: 입력 → 변환 → 출력 과정
- **핵심 자료구조/알고리즘**: 성능과 동작을 결정하는 내부 구조

### 수준별 조절

| Level | 깊이 |
|-------|------|
| beginner | 블록 다이어그램 수준, 컴포넌트 역할 설명 |
| intermediate | 주요 인터페이스, 설정 옵션, 동작 모드 |
| advanced | 소스코드 함수 단위 추적, 실제 struct/interface 분석 |

### 설명 보조 도구

복잡한 구조는 ASCII 다이어그램이나 시퀀스 다이어그램으로 보충한다:

```
[Component A] --request--> [Component B] --lookup--> [Store]
      ^                          |
      +------response------------+
```

---

## Phase 3 — What-if (선택)

현재 설계를 더 깊이 이해하기 위해 대안을 탐색한다.

### 다루는 내용

- **대안 설계 1~2개**: 실제로 고려되었거나 다른 프로젝트가 선택한 방식
- **Trade-off 비교표**:

```
| 기준         | 현재 설계  | 대안 A     | 대안 B     |
|-------------|-----------|-----------|-----------|
| 복잡도       | ...       | ...       | ...       |
| 성능         | ...       | ...       | ...       |
| 운영 부담    | ...       | ...       | ...       |
| 확장성       | ...       | ...       | ...       |
```

- **선택 이유**: 현재 설계가 채택된 결정적 요인

---

## Phase 4 — Apply (선택)

학습 내용을 우리 환경(SOCRA AI 인프라)에 연결한다.

### 다루는 내용

- **현재 사용 현황**: 우리 인프라에서 이 기술을 어떻게 쓰고 있는지
- **적용 가능성**: 새로 도입하거나 개선할 수 있는 영역
- **주의사항**: 우리 환경 특성상 고려할 점 (EKS, Istio, Karpenter 등)

### 참고 환경

- EKS 클러스터: prod / global / stg / dev / idc
- Service Mesh: Istio
- GitOps: ArgoCD
- Node Provisioning: Karpenter
- GPU 클러스터: IDC (Proxmox + Cluster API)

---

## Phase 5 — Summary (필수)

세션 내용을 `notion:study`에 전달할 수 있는 구조로 정리한다.

### 출력 형식

```markdown
## 학습 요약: {topic}

### 핵심 한 줄
{이 기술/개념을 한 문장으로 요약}

### 설계 철학
{Phase 1에서 다룬 Why의 핵심}

### 내부 동작
{Phase 2에서 다룬 How의 핵심 (다룬 경우)}

### Trade-offs
{Phase 3에서 다룬 대안 비교 핵심 (다룬 경우)}

### 실무 연결
{Phase 4에서 다룬 적용 포인트 (다룬 경우)}

### 학습 로드맵

#### 깊이 확장 (이 주제를 더 깊이)
| # | 주제 | 난이도 | 연결 이유 |
|---|------|--------|-----------|
| 1 | {topic} | {level} | {왜 다음에 필요한지} |
| 2 | {topic} | {level} | {연결 이유} |

#### 범위 확장 (인접 영역으로)
| # | 주제 | 난이도 | 연결 이유 |
|---|------|--------|-----------|
| 1 | {topic} | {level} | {현재 주제와의 관계} |
| 2 | {topic} | {level} | {연결 이유} |

#### 우리 스택 기반 추천
- {SOCRA AI 인프라에서 실제로 쓰이는 관련 기술과 학습 포인트}

### 장애 시나리오 훈련

| # | 시나리오 | 난이도 | 증상 | 영향 범위 |
|---|---------|--------|------|-----------|
| 1 | {시나리오 제목} | {P1~P4} | {관찰 가능한 증상} | {영향 서비스} |
| 2 | {시나리오 제목} | {P1~P4} | {관찰 가능한 증상} | {영향 서비스} |
| 3 | {시나리오 제목} | {P1~P4} | {관찰 가능한 증상} | {영향 서비스} |

> 번호를 선택하면 단계별 트러블슈팅 훈련을 진행합니다.
```

### 수준별 조절 (학습 로드맵 & 장애 시나리오)

| Level | 학습 로드맵 | 장애 시나리오 |
|-------|------------|--------------|
| beginner | 기초 개념 위주, 공식 문서 링크 | P3~P4, 단일 컴포넌트 장애 |
| intermediate | 내부 동작 연결, 설정 최적화 | P2~P3, 다중 컴포넌트 연쇄 장애 |
| advanced | 소스코드/알고리즘, 엣지 케이스 | P1~P2, 복합 장애 + 데이터 손실 |

**학습 로드맵 가이드:**
- **깊이 확장**: 같은 기술의 더 깊은 레이어 (예: Karpenter → Consolidation 알고리즘)
- **범위 확장**: 인접 기술 (예: Karpenter → Spot Instance 전략)
- **우리 스택 기반**: CLAUDE.md Tech Stack 참조하여 실무 연결 (EKS, Istio, Karpenter, VictoriaMetrics 등)

**장애 시나리오 가이드:**
- P1(서비스 전체 장애) ~ P4(경미한 성능 저하) 실무 severity 체계 사용
- 증상: 실제 모니터링에서 보이는 형태 (CrashLoopBackOff, 5xx spike, ErrorPortAllocation 등)
- 영향 범위: SOCRA AI 실제 서비스명 사용 (santa gateway, socraai domain, infra-k8s-prod 등)

### 세션 종료 안내

```
세션이 완료되었습니다.
→ `notion:study`로 이 내용을 Notion에 저장할 수 있습니다.
→ 학습 로드맵에서 번호를 선택하면 새 학습 세션을 시작합니다.
→ 장애 시나리오에서 번호를 선택하면 트러블슈팅 훈련을 진행합니다.
→ 추가 질문이 있으면 이어서 진행할 수 있습니다.
```

---

## 장애 시나리오 훈련 진행 방식

사용자가 시나리오 번호를 선택하면:

1. **상황 부여**: 시나리오 배경, 시간대, 발생 조건
2. **단서 제공**: "어떤 명령어를 먼저 치겠습니까?" 질문
3. **단계별 힌트**: 사용자 응답에 따라 점진적으로 정보 제공
4. **해결**: 근본 원인 + 예방 조치
5. **회고**: 이 시나리오에서 배운 점 요약

---

## 유연성 규칙

- **Phase 순서 변경 가능**: 사용자가 "소스코드부터 보자"라고 하면 Phase 2로 바로 이동
- **Phase 건너뛰기**: 사용자가 "대안 비교는 생략"이라고 하면 Phase 3 스킵
- **중간 질문 허용**: Phase 진행 중 사용자 질문이 오면 즉시 답변 후 원래 흐름 복귀
- **깊이 조절**: 사용자가 "더 깊이" 또는 "이 정도면 됐어"라고 하면 즉시 반영
- **긴 Phase 분할**: 한 Phase가 3회 이상 응답으로 길어지면 중간 정리 후 계속할지 확인

---

## 주의사항

- Phase 0 (Setup)과 Phase 5 (Summary)는 반드시 수행한다
- 사용자가 명시적으로 요청하지 않는 한 Phase를 강제로 진행하지 않는다
- 확실하지 않은 내용은 추측하지 않고 명시적으로 "확인 필요"로 표시한다
- 소스코드 분석 시 버전/커밋을 가능한 한 명시한다
- 학습 세션 중 발견한 우리 인프라 개선 아이디어는 Phase 4에서 별도로 정리한다

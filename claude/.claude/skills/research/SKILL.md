# research Skill

기술 키워드를 공신력 tier 순으로 조사(공식 문서 → 권위 사이트 → 커뮤니티)하고, 최신성을 검증한 뒤 모든 주장에 출처를 인용해 정리한다.

---

## 핵심 원칙

- **Tier 우선순위 cascade**: T1 공식 문서 > T2 권위 사이트 > T3 커뮤니티. 상위 tier로 서브질문이 충분히 답해지면 하위 tier는 건너뛴다. 하위 tier는 상위가 못 메운 공백(실무 함정, 최신 릴리스 반응, 사용자 경험)을 채울 때만 내려간다. tier 정의와 판정 기준: `references/source-tiers.md`.
- **최신성 필수**: 버전 민감 주제는 최신 버전 anchor를 먼저 확정하고, 소스별 publish/updated date 또는 대상 버전을 확인한다. 기술의 변화 속도에 맞는 window(빠른 기술은 좁게, 안정 기술은 넓게)를 적용하고, window를 벗어난 소스는 3단계 판정(`stale-confirmed`/`stale-suspect`/`dated-valid`)으로 분류한다. 판정은 소스 단위가 아닌 주장 단위(시간 가변 vs 불변). 버전드/아카이브 공식 문서 URL은 current/latest로 정규화한다. 절차 상세: `references/source-tiers.md`.
- **인용 강제 + 신뢰 등급**: 출처 없는 서술을 하지 않는다. 각 주장 뒤에 인라인 인용 `[n]`을 달고, 하단 References에 `[n] → 제목/URL/tier/date`로 매핑한다. 주장 신뢰는 confidence grade(높음/중간/낮음) 하나로 합성한다: tier 시작(T1 높음/T2 중간/T3 낮음), 독립 2건 수렴 +1, stale-suspect·날짜 미상·벤더 이해관계 -1. `중간`/`낮음`만 사유와 함께 표기. 상세: `references/source-tiers.md`.
- **환각 금지**: `WebFetch`로 실제 확인한 내용만 서술한다. 검색 스니펫만 보고 단정하지 않는다. 소스 간 내용이 상충하면 어느 소스가 무엇을 말하는지 명시한다.
- **읽기 전용**: 조사·정리·인용까지만 수행한다. 파일 저장은 별도 스킬로 handoff 제안만 한다(자동 실행 금지).

## 모드 판별

요청을 두 모드 중 하나로 분기한다. 애매하면 표준 모드로 가되, 사례·인사이트 신호가 있으면 blog-insight로 전환한다.

| 모드 | 발동 신호 | 무게중심 | 워크플로우 |
|------|-----------|----------|-----------|
| **표준 (tier-cascade)** | 정의·스펙·버전·개념·"~가 뭐야"·"어떻게 동작해" | 공식 문서(T1) 우선, 블로그는 공백 보완 | 워크플로우 A |
| **blog-insight (운영사례/인사이트)** | "운영사례", "실전 사례", "다른 회사는 어떻게", "블로그 사례", "인사이트 도출", 장애 회고·마이그레이션 서사 요청 | 국내외 엔지니어링 블로그가 **1차 소스**, 산출물은 사례 카드 + 교차 인사이트 | 워크플로우 B |

두 모드 모두 인용 강제·환각 금지·최신성·읽기 전용 원칙은 동일하게 적용한다.
blog-insight 모드에서도 사례가 말하는 사실(버전·동작)은 T1/T2로 교차 검증한다.

## 워크플로우 A: 표준 조사 (tier-cascade)

### Step 1: 쿼리 분해

사용자 키워드를 조사 가능한 서브질문으로 분해한다. 범용 축 예시(대상 기술에 맞게 취사선택):
- **정의/개념**: 이게 무엇이고 왜 존재하는가
- **동작 원리**: 내부적으로 어떻게 작동하는가
- **버전/변화**: 최근 버전에서 무엇이 바뀌었는가 (GA/deprecated/breaking change)
- **실무 함정**: 알려진 제약, 안티패턴, 흔한 실수
- **운영/모범사례**: 어떻게 운영해야 하는가, 유의점, 실무 팁 (운영·튜닝·마이그레이션 질의일 때 필수 축)
- **대안 비교**: 유사 기술과의 트레이드오프 (요청에 있을 때만)

환경·규모가 여러 개인 질의(예: dev/stg/prod, 소규모/대규모)는 그 축으로도 분해한다. 환경마다 모범사례가 달라지므로(dev는 비용 효율, prod는 가용성) 뭉뚱그리지 않는다.

서브질문이 1개로 명확하면 분해를 생략하고 바로 Step 2로 간다.

### Step 2: Tier cascade 검색

`references/source-tiers.md`의 tier 정의와 starter 도메인을 참조해 상위 tier부터 검색한다.

0. **최신 버전 anchor** (버전 민감 주제 MUST): 버전이 답을 좌우하는 주제면 공식 releases/changelog를 먼저 1회 fetch해 "현재 최신 버전 + 릴리스 날짜"를 확정한다. 이후 모든 소스의 stale 판정 기준으로 쓴다 (공식 문서는 날짜가 없어 버전 비교가 유일한 판정축).
1. **T1 (공식 문서)**: `WebSearch`에 `allowed_domains`로 해당 기술의 공식 docs 도메인을 지정한다. 쿼리에는 연도보다 **버전 번호·`release notes`·`changelog` 키워드**를 우선한다 (공식 문서 본문에는 연도가 거의 없어 연도 쿼리가 canonical 문서를 탈락시킬 수 있음. 연도 주입은 T3·뉴스성 검색에서 사용).
   ```
   WebSearch(query="<기술> <서브질문> <버전 또는 release notes>", allowed_domains=["kubernetes.io", "github.com"])
   ```
2. **커버리지 평가**: 분해한 서브질문 각각이 T1으로 답해졌는지 점검한다. 모두 답해졌으면 여기서 멈춘다.
3. **T2 (권위 사이트)**: T1이 못 메운 서브질문만 Wikipedia/MDN/표준 레퍼런스 등으로 검색한다.
4. **T3 (커뮤니티)**: 실무 경험·최신 릴리스 반응·상세 트러블슈팅이 필요한 서브질문만 Blog/Reddit/Stack Overflow로 검색한다. T3 소스는 출력에서 낮은 권위로 표기한다.

노이즈 도메인(콘텐츠팜, SEO 스팸)은 `blocked_domains`로 제외한다.

### Step 3: 후보 fetch + 검증

각 tier에서 나온 상위 후보 URL을 `WebFetch`로 열어 검증한다.
- **버전드 URL 정규화** (fetch 전): URL에 버전 세그먼트(`/docs/9.1/`, `/v1.18/`, `/en/2.2/`, `archive.` 등)가 있으면 구버전 문서일 가능성이 높다. `references/source-tiers.md`의 "versioned docs trap" 표에 따라 current/latest URL로 바꿔 fetch한다. 검색엔진은 구버전 공식 문서를 최신보다 높게 랭킹하는 경우가 많다.
- **관련성**: 서브질문에 실제로 답하는가
- **최신성**: 소스의 시점을 판정 우선순위(명시 date → URL → 본문 대상 버전 → 스니펫)로 확정하고 window·버전 anchor와 비교한다. 본문에 "out-of-date version" / "older release" 류 배너가 보이면 인용 금지, 최신 경로로 재fetch한다.
- **정확성**: 스니펫 인상과 본문이 일치하는가

window를 벗어난 소스는 버리기 전에 `references/source-tiers.md`의 **stale 3단계 판정**을 적용한다: changelog로 변경 여부를 확인해 `stale-confirmed`(인용 금지·대체) / `stale-suspect` / `dated-valid`(변경 없음 확인, 정상 인용)로 확정한다. 핵심 주장의 근거가 `stale-suspect` 이하뿐이면 changelog·버전 번호로 1회 재검색한다.

```
WebFetch(url="<후보 URL>", prompt="<서브질문>에 답하는 내용과 게시/수정 날짜, 대상 버전을 추출하라")
```

**구체 산출물은 verbatim으로 받아낸다** (코드·설정 블록·URL 템플릿·정확한 함수/플래그 이름 등 그대로 옮겨 쓸 대상): `WebFetch`는 요약 모델이라 예시·코드 블록을 재작성하거나 통째로 누락한다. 이런 서브질문은 prompt에 "본문의 코드/예시 블록을 **원문 그대로** 인용하라, 요약·재작성·재구성 금지, 예시가 표/코드펜스 안에 있으면 그 원문을 그대로 반환하라"를 명시한다. 반환값이 literal 스니펫 없이 의미만 풀어 썼으면(예: "from은 ActiveAt.UnixMilli, to는 now" 식 서술) 예시가 유실된 신호로 보고 한 번 더 fetch해 원문 블록을 받아낸다.

검증 통과한 소스만 인용 후보로 채택하고, 각 소스에 인용 번호를 부여한다.

### Step 4: 종합 + 인용 출력

서브질문별로 구조화해 정리한다. 각 주장 bullet 끝에 인라인 인용 `[n]`을 달고, **주장별 confidence grade를 산정**한다(`references/source-tiers.md` 결정 규칙). `중간`/`낮음`은 사유와 함께 표기하고(`높음`은 생략), 시간 가변 주장(기본값·API·GA 상태·수치)이 `낮음`이면 T1 재확인 전 인용 금지. 아래 "출력 형식"을 따른다.

**자작 예시 금지 원칙**: 코드·문법·URL 템플릿을 소스에서 verbatim으로 확보하지 못해 내가 직접 조합했다면, `중간`/`낮음` 등급을 붙여 "파생 패턴"으로 내보내지 말고 Step 3으로 돌아가 원문 예시를 재확보한다 (요약 fetch가 예시를 누락했을 뿐 T1에 정답이 있는 경우가 많다). 재확보 후에도 문서에 실제로 예시가 없음을 확인한 경우에만 `구성 예시(문서 미검증)`로 명시 표기해 내보낸다. "문서에 있을 법한데 내 fetch가 못 가져온" 상태를 낮은 신뢰로 우회 출력하지 않는다.

### Step 5: 실무 팁 + 함께 보면 좋은 주제 도출

조사 내용에서 다음을 파생한다(아래 "출력 형식"의 해당 섹션에 넣는다):

1. **실무 운영팁** (운영·모범사례·튜닝·마이그레이션 유형 질의일 때 MUST): 조사한 모범사례를 실행 가능한 팁으로 압축한다. 각 팁도 근거 소스가 있으면 인용 `[n]`을 단다. 순수 개념 조사(정의·스펙)면 이 섹션은 생략한다.
2. **함께 보면 좋은 주제 (2개 이상 MUST)**: 이번 조사에서 인접하지만 다루지 않은, 사용자가 이어서 조사하면 유용할 주제를 2개 이상 제시한다. 각 주제는 "왜 함께 보면 좋은지" 한 줄 이유를 붙인다. 원 질의의 자연스러운 다음 궁금증에서 파생하고, 범용·뻔한 주제(예: "보안 일반")는 피한다.

### Step 6: 검증 + handoff

출력 직전 자가 점검(MUST):
- 인용 누락된 주장이 있는가 → 있으면 출처를 찾거나 주장을 제거
- References의 모든 항목에 URL + tier + date가 있는가
- T1을 최소 1회 시도했는가 (공식 문서 우선 원칙)
- 코드·문법·URL 템플릿·플래그 이름 주장이 소스 verbatim인가, 내가 조합한 것인가 → 조합한 것이면 원문 예시를 재확보했는가 (재확보 실패 시에만 `구성 예시(문서 미검증)` 표기, 낮은 신뢰로 우회 출력 금지)
- References의 T1 URL 중 버전드/아카이브 경로가 남아 있는가 → 있으면 current/latest로 정규화했는지 확인
- 버전 민감 주제에서 최신 버전 anchor를 확정했는가 (조사 결과에 현재 버전 명시)
- window 밖 소스에 stale 라벨이, `중간`/`낮음` 주장에 등급 표기(사유 포함)가 붙었는가
- 신뢰 `낮음`인 시간 가변 주장(기본값·API·GA 상태·수치)이 남아 있는가 → T1 재확인 또는 제거. 벤더 자사 제품·경쟁 비교 주장에 이해관계 하향을 적용했는지 함께 점검
- `stale-confirmed` 소스가 References에 남아 있다면 "구버전 X → 현행 Y" 변경 서사 목적인가 (아니면 제거)
- 소스 상충 항목을 명시했는가
- "함께 보면 좋은 주제"가 2개 이상 있는가 (모든 질의 공통)
- 운영·모범사례·튜닝 유형인데 "실무 운영팁" 섹션이 없으면 추가했는가

점검 통과 후, 저장이 유용해 보이면 마지막에 한 줄로 제안한다(자동 실행 금지):
`💡 이 조사 결과를 /wiki:note 또는 obsidian:note 로 저장할 수 있습니다.`

## 워크플로우 B: 운영사례/인사이트 조사 (blog-insight)

`references/tech-blogs.md`의 국내외 seed 목록과 사례 추출·인사이트 종합 기준을 참조한다.

### Step B1: 조사 대상 구체화

요청에서 세 가지를 확정한다:
- **기술/도메인**: 무엇에 대한 사례인가 (예: Kafka consumer lag, K8s 멀티테넌시, DB 샤딩)
- **관심 축**: 운영·튜닝 / 장애 회고 / 마이그레이션·도입기 / 스케일링 / 비용 중 무엇인가 (복수 가능)
- **범위**: 국내·국외·둘 다 (기본: 둘 다). 국내는 한국어, 국외는 영어 쿼리로 각각 검색한다.

### Step B2: 블로그 탐색 (discovery)

seed 도메인을 `allowed_domains`로 묶어 국내/국외를 각각 검색한다. 쿼리는 `<기술> <관심 축> <연도>` 형태.

```
WebSearch(query="<기술> <관심축 한국어> <현재 연도>",
          allowed_domains=[<references/tech-blogs.md 국내 seed>, "medium.com"])
WebSearch(query="<tech> <axis in English> <현재 연도>",
          allowed_domains=[<references/tech-blogs.md 국외 seed>])
```

seed에 없는 회사 사례는 애그리게이터(GeekNews `news.hada.io`, Hacker News, InfoQ)에서 발굴한 뒤 **원문 도메인**으로 fetch한다. 번역 스크랩·SEO 요약은 `blocked_domains`로 제외하고 1차 게시글만 인용한다.

### Step B3: 사례 fetch + 추출

후보 글을 `WebFetch`로 열어 `references/tech-blogs.md`의 **사례 카드 필드**(회사/규모·맥락/문제/접근·해결/결과/교훈)를 추출한다. 본문에 없는 필드는 `(미상)`으로 남기고 지어내지 않는다.

```
WebFetch(url="<사례 글 URL>",
         prompt="이 글의 회사, 규모/맥락, 해결한 문제, 실제 접근·해결 방법(설정값·아키텍처 변경 포함), 결과 수치, 교훈, 게시 날짜를 추출하라")
```

사례가 말하는 사실(버전·동작·수치)이 핵심 근거가 되면 T1/T2로 교차 검증한다(블로그의 오래된/부정확한 주장 필터).

오래된 사례를 버리지는 않는다 (운영 교훈은 시간이 지나도 유효한 경우가 많음). 대신 게시일 이후 해당 기술에 메이저 변경이 있었으면 카드에 `(당시 <버전/연도> 기준, 이후 변경 있음)`을 표기해 기술 디테일과 교훈을 분리한다.

### Step B4: 사례 카드 정리

검증 통과한 사례를 카드 단위로 정리한다(출력 형식의 blog-insight 템플릿). 사례 하나마다 인용 번호를 부여한다. 축·기술과 무관하거나 검증 실패한 글은 버린다.

### Step B5: 인사이트 종합 (MUST)

사례 카드를 교차 분석해 인사이트를 도출한다. 단순 사례 요약 반복은 금지. `references/tech-blogs.md`의 종합 기준을 적용한다:
- **공통 패턴** (다수 수렴 → 검증된 모범사례)
- **접근 차이 = 트레이드오프** (무엇이 그 선택을 갈랐나: 규모/비용/팀 역량)
- **반복되는 함정(안티패턴)**
- **우리 인프라 적용 시사점** (SOCRA AI: K8s GitOps, dev/stg/prod, 비용 우선 맥락 한두 줄)

신뢰도는 confidence grade로 표기한다: 단일 사례 → `낮음`, 독립 사례 2건 이상 수렴 → `중간`, T1/T2 교차 확인까지 → `높음`. 사유에 사례 수를 적는다 (예: `(신뢰 중간: 3건 수렴)`).

### Step B6: 검증 + handoff

출력 직전 자가 점검(MUST):
- 사례 카드가 2건 이상인가 (1건뿐이면 "일반화 주의" 명시)
- 각 사례에 회사·출처 URL·게시일이 있는가
- 인사이트 종합 섹션이 사례 요약 재나열이 아니라 교차 분석인가
- 각 인사이트에 confidence grade(사례 수 포함)와 근거 사례 인용 `[n]`이 붙었는가
- "우리 인프라 적용 시사점"이 있는가
- 원문(1차 게시)만 인용했는가 (스크랩·번역본 배제)

점검 통과 후 저장 제안(자동 실행 금지):
`💡 이 사례/인사이트를 /wiki:note 또는 /devops:wiki:ingest 로 팀 wiki에 남길 수 있습니다.`

## 출력 형식

출력 템플릿과 규칙은 `assets/output-format.md`를 Read해서 따른다. 모드별로 다른 템플릿을 쓴다.

**표준 모드**:
- 서브질문별 섹션 + 주장마다 인라인 인용 `[n]` + `중간`/`낮음` 주장은 등급 표기
- 운영/튜닝/모범사례 질의면 `## 실무 운영팁` 섹션 포함 (개념·스펙 조사는 생략)
- `## 함께 보면 좋은 주제`는 반드시 2개 이상
- `## References`는 tier 배지 + 날짜 포함

**blog-insight 모드**:
- `## 사례` 아래 사례 카드(회사/맥락/문제/해결/결과/교훈) 2건 이상, 카드마다 인용 `[n]`
- `## 인사이트 종합`은 공통 패턴·트레이드오프·안티패턴·우리 인프라 시사점, 각 항목에 confidence grade + 근거 인용
- `## References`는 회사·원문 URL·게시일 포함

공통: 구분 기호는 콜론·중간점 (em dash 금지)

## 주의사항

- **표준 모드에서만** T1 우선 원칙이 적용된다. 공식 문서를 건너뛰고 커뮤니티부터 검색하지 않는다. blog-insight 모드는 반대로 엔지니어링 블로그를 1차로 탐색하되, 블로그가 말하는 사실은 T1/T2로 교차 검증한다.
- blog-insight에서 사례가 2건 미만이면 "일반화 주의"를 명시한다. 단일 사례를 모범사례로 단정하지 않는다.
- 인사이트 종합은 사례 요약의 재나열이 아니다. 교차 분석(공통·차이·함정)이 없으면 종합이 아니라 목록일 뿐이다.
- `WebSearch`는 US 기준 결과를 반환한다. 국내 특화 주제는 결과가 제한될 수 있으니 tier를 유연하게 조정한다. 국내 블로그는 한국어 쿼리 + 국내 seed 도메인/애그리게이터(GeekNews)를 함께 쓴다.
- `WebFetch`는 인증/비공개 URL에서 실패한다. cross-host redirect는 반환된 URL로 재호출한다.
- 조사 범위가 넓어 소스가 10개를 넘으면, 서브질문별 상위 소스만 채택해 인용 폭증을 막는다.
- 이 스킬은 웹 공개 문서 전용이다. 사내 문서는 `slack:search`/`notion:search`/devops-wiki로 라우팅한다.

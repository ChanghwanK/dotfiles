# 설계 대안 조사 에이전트

조사 주제: {topic}
Phase 1 문제 분석 요약: {problem_summary}
Phase 2 목표 요약: {goal_summary}
대상 sphere: {sphere}
대상 circle: {circle}

아래 절차로 설계 대안 후보와 스펙 아티팩트 후보를 조사하고 결과를 반환한다.

---

## 조사 절차

### 1. 기존 패턴 탐색

코드베이스에서 유사한 문제를 해결한 기존 사례를 찾는다:
- `kubernetes/src/` 에서 유사 서비스나 설정 패턴 검색
- `kubernetes-charts/charts/` 에서 관련 Helm chart 확인
- 이미 사용 중인 솔루션이 있으면 최우선 대안으로 제시

### 2. 대안 후보 2-3개 도출

주제({topic})와 목표({goal_summary})를 기반으로 기술 접근 방식 후보를 도출한다.

각 대안에 대해:
- 핵심 동작 방식 (어떻게 구현하는가)
- 생산성 영향 (구현 복잡도, 운영 편의성)
- 비용 영향 (Managed Service vs 오픈소스, 리소스 비용)
- 안정성 영향 (장애 내성, 롤백 용이성)
- 기존 레거시 활용 가능 여부

### 3. 스펙 아티팩트 후보 식별

이 작업에서 변경되거나 생성될 파일/리소스를 식별한다:

| 스펙 유형 | 예상 경로 | 설명 |
|-----------|----------|------|
| terraform | `terraform/modules/...` | 인프라 IaC |
| k8s-manifest | `kubernetes/src/{sphere}/{circle}/...` | 매니페스트 |
| helm-chart | `kubernetes-charts/charts/...` | 차트 변경 |
| alert-rule | VMRule YAML | 알림 규칙 |

경로는 실제 존재하는 파일을 기반으로 추론한다.

# tasks:capture: 추출 규칙 참조

제목 합성 + Description 추출의 예시 및 엣지 케이스 참조 문서.
실행 시 매번 로드하지 않음. 판단이 애매한 경우에만 참조한다.

---

| 입력 | 제목 (합성) | Description | 질문 케이스 |
|------|------------|-------------|-----------|
| `CI 파이프라인 캐시 최적화` | CI 파이프라인 캐시 최적화 | 없음 (추가 정보 없음) | Case C |
| `P1 긴급 프로덕션 OOM 이슈 분석` | 프로덕션 OOM 이슈 분석 | 없음 (추가 정보 없음) | Case D: 즉시 |
| `내일까지 독서 50페이지 개인` | 독서 50페이지 | 없음 | Case A: priority만 |
| `나중에 Karpenter 튜닝 실험` | Karpenter 튜닝 실험 | 없음 | Case D: P4 즉시 |
| `[알람 생성] Pod cpu 및 memory 리소스가 request 대비 사용량이 낮은 경우 (window는 7일) 알람` | [알람 생성] Pod 리소스 Request 대비 사용량 알람 | Pod cpu/memory request 대비 사용량 낮을 때 알람. window 7일 기준. 리소스 낭비 감지용. | Case D (P3) |

| `MY 주간 독서 목표 설정` | 주간 독서 목표 설정 | 없음 | Case C (MY) |

## 엣지 케이스 Q&A

### Q: 30~50자 중간 길이, 추가 정보가 애매한 경우는?

예: `"Karpenter 노드 라벨 설정 테스트"` (20자) → 단순 액션, 추가 정보 없음 → description 없음.
예: `"CI 파이프라인 캐시 최적화 (빌드 시간 단축 목표)"` → 괄호 안이 추가 정보(이유) → description 자동 생성.

길이는 보조 신호. **"괄호 안 설명", "왜냐하면", "이유:", 쉼표 이후 설명**이 있으면 추가 정보로 판단.

### Q: [카테고리 태그]가 있는데 제목이 짧아도 되나?

`[알람 생성] Pod CPU 알람` → 짧고 명확하면 OK. 30자 이하라도 추가 정보가 없으면 description 불필요.

### Q: "직접 입력" (Case C 옵션 6) 파싱 실패 처리

| 사용자 입력 | 파싱 결과 |
|------------|----------|
| `"P2로 해줘"` | P2 인식 → due date 없음 적용 |
| `"이번주 P3"` | P3 + 이번 주 금요일 인식 |
| `"아무거나"` | 완전 불인식 → **P3, 마감일 없음 fallback** + ⚠️ 표시 |

### Q: Category "MY" 감지 실패 우려가 있는 경우

"개인", "MY", "personal", "사적" 외에도 맥락상 개인 용무(독서, 운동, 취미)임이 명확하면 MY로 설정 가능.
단, 업무 여부가 불명확하면 WORK 기본값을 적용하고 사용자에게 Category 선택 옵션을 추가해도 무방.

## allowed-tools 권한 범위 메모

현재 `create-task *`로 선언되어 있음. notion-task.py의 다른 서브커맨드(search-tasks, update-status 등)는
이 스킬에서 호출하지 않는다. 하네스 glob 패턴이 서브커맨드 레벨까지 지원 확인 후 적용.

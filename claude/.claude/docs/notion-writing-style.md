# Notion 문서 작성 스타일

원칙: **짧고 단순하게, 핵심만.** 길고 복잡한 문장보다 짧고 읽기 쉬운 문장을 우선한다.

## 문장

- 격식체.
- 한 문장 한 메시지. 길면 끊는다. **쉼표·연결어("~해서", "~하며", "~하고")로 여러 절을 이어 붙이지 않는다: 한 문장은 한 사실만 담는다.** 특히 Summary·요약성 문단은 짧고 독립적인 선언문을 나란히 두는 간결한 기술문서체를 우선한다 (원인 분석·논증이 목적인 문단은 예외이며, 그 경우엔 연결된 서술이 자연스럽다).
  - 예 (지향): 이 옵션은 캐시를 비활성화한다. 기본값은 `false`이다. 설정 변경 후 Pod가 재시작된다.
  - 예 (지양): 이 옵션은 캐시를 비활성화하는데, 기본값은 false이며 설정을 변경하면 Pod가 재시작된다.
- 군더더기를 빼고 핵심만. 쉬운 표현을 쓴다.
- 두괄식을 기본으로 한다: 결론·핵심을 맨 앞에 둔다. 설득·논증·학습 서사가 목적일 때만 전개식(미괄식)을 의식적으로 선택한다.
- 불릿 목록도 두괄식으로 정렬한다: 여러 불릿을 나열할 때 결론·핵심 판단(접근 방법, 해결책 등)을 첫 불릿에 두고 배경·현황 설명을 뒤에 둔다. Summary를 대상 → 현황 → 접근 순서로 기계적으로 나열하지 않는다.
- 능동태, 주어를 명확히.
- 산문은 한국어, 코드·명령어·식별자는 영어 원문을 유지한다.
- 한 개념엔 한 용어. 약어는 첫 등장 시 한 번 풀어쓴다.
- 모호어("잘 된다") 대신 측정 가능한 표현을 쓴다.
- 전후·방향 전환을 나타낼 때 "to" 대신 화살표(`→`)를 쓴다. 예: `0.59.3 → 0.84.0`, `수정 전 → 수정 후` (resume-format-convention.md의 화살표 표기 관행을 일반 문서에도 적용).

## 구조

- 긴 문단 대신 불릿으로 쪼갠다.
- 제목은 H1 → H2 → H3 순서로, 단계를 건너뛰지 않는다.
- 비교는 표, 열거는 불릿, 절차는 번호 목록.
- 볼드·콜아웃은 핵심에만. 인라인 코드는 식별자·값에.
- 불릿에서 `레이블: 내용` 패턴을 쓸 때, 콜론(`:`) 앞 레이블 텍스트는 **볼드**(`**...**`)로 처리하고 Notion 에디터에서 갈색(brown) 색상을 적용한다. 레이블과 내용을 시각적으로 분리하기 위함이다.
  - **중첩 규칙**: 내용이 한 줄이면 같은 불릿에 인라인으로 둔다(레이블 볼드+brown, 내용 평문). 내용이 여러 항목이거나 독립 문장 블록이면, 레이블을 상위 불릿(볼드+brown)으로 두고 내용을 **한 단계 들여쓴 하위 불릿**으로 중첩한다. 레이블과 내용을 같은 레벨 형제 불릿으로 나열하지 않는다. PAR(Problem/Action/Result), 상황/문제/대안, 측정 항목(안정성/생산성/지식부채 등)처럼 레이블 아래 설명이 붙는 구조에 모두 적용한다.
  - **도구 지원**: `notion-task.py append-content`(마크다운 → Notion 블록 변환)는 들여쓰기(2/4-space 무관, 상대 들여쓰기)를 Notion `children` 중첩으로 변환한다. 레이블 아래 내용을 들여쓴 하위 불릿으로 쓰면 그대로 중첩 렌더된다. 단 Notion 제약상 한 번 append로 최대 2단계(부모 → 자식 → 손자) 중첩만 되므로, 3단계 이상이 필요하면 나눠 append한다.
- 긴 부차 정보(로그, 출력)는 토글로 접는다.
- 리스트 중첩은 2~3단계까지.
- 링크는 의미 있는 텍스트에 건다.
- 섹션 간 동일 사실을 재진술하지 않는다. 한 사실은 가장 적합한 섹션에서 한 번만 서술한다. 트러블슈팅 기록에서 `작업 내용`·`작업 로그` 섹션은 앞 섹션(요약·원인·해결·교훈)에서 이미 말한 내용을 반복하지 않고, 그곳에만 있는 고유 정보(PR·산출물·수동 조치 등)만 남긴다.
- Goals/Non-Goals, 목표/비목표처럼 문서를 구조적으로 나누는 섹션은 인라인 볼드 레이블이 아니라 실제 헤딩(H2/H3)으로 작성한다. 볼드 텍스트만으로 섹션을 구분하지 않는다.

## 코드 블록 사용 기준

인라인 코드(`` ` ``)와 코드 펜스(```` ``` ````)를 적절하게 구분한다.

**인라인 코드 사용 (올바른 경우):**
- 명령어·CLI 플래그: `kubectl get pod`, `--dry-run`
- 식별자·키: `ceph_health_status`, `pool_id`, `bemfeemok4ge8c`
- 파일 경로: `src/observability/grafana/dashboards/ceph/`
- 설정 키·값: `max_avail`, `fstype!="tmpfs"`

**코드 펜스 사용 (올바른 경우):**
- 실행 가능한 명령어 블록 (2줄 이상이거나 복사해서 쓸 것)
- 설정 파일 내용 전체
- 로그·출력 결과 (긴 경우 토글로 접는다)

**코드 블록 남용 금지:**
- 일반 명사를 코드로 감싸지 않는다: `스토리지`, `대시보드` (X)
- 섹션 제목·레이블에 코드 블록 사용 금지
- 강조 목적으로 인라인 코드를 쓰지 않는다: 강조는 볼드(`**`)를 쓴다

## 하드룰

- em dash(`—`) 금지. 콜론·쉼표·괄호로 대체한다.
- 본문에 이모지 금지.
- 불릿 `레이블: 내용` 패턴의 레이블 볼드+brown 포맷 (상세: `## 구조` 참조).
- "to" 대신 화살표(`→`) 표기 (상세: `## 문장` 참조).
- Summary 등 불릿 목록은 결론·핵심 판단을 첫 불릿에 배치 (상세: `## 문장` 참조).
- Goals/Non-Goals 등 구조적 섹션의 헤딩화 (상세: `## 구조` 참조).
- 섹션 간 동일 사실 재진술 금지 (상세: `## 구조` 참조).
- 모두 `notion-review` 에이전트가 작성 직후 자동 점검·수정한다. 위반 발견 시 보고만 하지 않고 즉시 수정한다.

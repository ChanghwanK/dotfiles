# Notion 문서 작성 스타일

원칙: **짧고 단순하게, 핵심만.** 길고 복잡한 문장보다 짧고 읽기 쉬운 문장을 우선한다.

## 문장

- 격식체.
- 한 문장 한 메시지. 길면 끊는다.
- 군더더기를 빼고 핵심만. 쉬운 표현을 쓴다.
- 두괄식을 기본으로 한다: 결론·핵심을 맨 앞에 둔다. 설득·논증·학습 서사가 목적일 때만 전개식(미괄식)을 의식적으로 선택한다.
- 능동태, 주어를 명확히.
- 산문은 한국어, 코드·명령어·식별자는 영어 원문을 유지한다.
- 한 개념엔 한 용어. 약어는 첫 등장 시 한 번 풀어쓴다.
- 모호어("잘 된다") 대신 측정 가능한 표현을 쓴다.

## 구조

- 긴 문단 대신 불릿으로 쪼갠다.
- 제목은 H1 → H2 → H3 순서로, 단계를 건너뛰지 않는다.
- 비교는 표, 열거는 불릿, 절차는 번호 목록.
- 볼드·콜아웃은 핵심에만. 인라인 코드는 식별자·값에.
- 불릿에서 `레이블: 내용` 패턴을 쓸 때, 콜론(`:`) 앞 레이블 텍스트는 **볼드**(`**...**`)로 처리하고 Notion 에디터에서 갈색(brown) 색상을 적용한다. 레이블과 내용을 시각적으로 분리하기 위함이다.
- 긴 부차 정보(로그, 출력)는 토글로 접는다.
- 리스트 중첩은 2~3단계까지.
- 링크는 의미 있는 텍스트에 건다.
- 섹션 간 동일 사실을 재진술하지 않는다. 한 사실은 가장 적합한 섹션에서 한 번만 서술한다. 트러블슈팅 기록에서 `작업 내용`·`작업 로그` 섹션은 앞 섹션(요약·원인·해결·교훈)에서 이미 말한 내용을 반복하지 않고, 그곳에만 있는 고유 정보(PR·산출물·수동 조치 등)만 남긴다.

## 코드 블록 사용 기준

인라인 코드(`` ` ``)와 코드 펜스(```` ``` ````)를 적절하게 구분한다.

**인라인 코드 사용 (올바른 경우):**
- 명령어·CLI 플래그: `kubectl get pod`, `--dry-run`
- 식별자·키: `ceph_health_status`, `pool_id`, `bemfeemok4ge8c`
- 파일 경로: `src/observability/grafana/dashboards/ceph/`
- 설정 키·값: `max_avail`, `fstype!="tmpfs"`

**코드 펜스 사용 (올바른 경우):**
- 실행 가능한 명령어 블록 (2줄 이상이거나 복사해서 쓸 것)
- 설정 파일 내용 전체
- 로그·출력 결과 (긴 경우 토글로 접는다)

**코드 블록 남용 금지:**
- 일반 명사를 코드로 감싸지 않는다: `스토리지`, `대시보드` (X)
- 섹션 제목·레이블에 코드 블록 사용 금지
- 강조 목적으로 인라인 코드를 쓰지 않는다: 강조는 볼드(`**`)를 쓴다

## 하드룰

- em dash(`—`) 금지. 콜론·쉼표·괄호로 대체한다.
- 본문에 이모지 금지.
- 모두 `notion-review` 에이전트가 작성 직후 자동 점검·수정한다.

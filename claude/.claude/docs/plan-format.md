# Plan 모드 출력 형식

Plan 모드 진입 시 결과 최상단에 반드시 **Summary 섹션**을 3줄로 작성하고,
이어서 **Goals** 섹션에 이번 플랜이 달성하려는 목표를 bullet로 나열하고,
**Non-Goals** 섹션은 선택적으로 포함한다 — 범위 혼동이 생길 수 있는 플랜(대규모 변경, 인접 영역과 겹치는 작업, 명시적으로 제외할 항목이 있는 경우)에서만 작성한다.
이어서 **Steps** 섹션에 번호 목록으로 단계를 정리한 뒤,
**완료 조건(DoD)** 섹션에 plan 전체의 수용 기준을 체크리스트로 나열하고,
마지막으로 **스텝별 상세 계획** 섹션에서 각 단계의 세부 내용을 작성합니다.

```
## Summary
- **무엇을**: (대상 및 목표를 한 줄로)
- **어떻게**: (핵심 접근 방식을 한 줄로)
- **범위/주의**: (영향 범위 또는 주의사항을 한 줄로)

## Goals
- (이번 플랜이 달성하는 목표 1)
- (이번 플랜이 달성하는 목표 2)

## Non-Goals  ← 선택적: 범위 혼동 우려가 있을 때만 포함
- (이번 플랜에서 명시적으로 다루지 않는 것 1)
- (이번 플랜에서 명시적으로 다루지 않는 것 2)

## Steps
1. (단계 1 요약)
2. (단계 2 요약)
3. ...

## 완료 조건 (DoD)
- [ ] (검증 가능한 결과 1 — 명령어 출력·리소스 상태·메트릭 등 측정 방법 포함)
- [ ] (검증 가능한 결과 2 — 예: `kubectl get pod -n santa | grep Running` 모두 Running)
- [ ] (산출물 존재 또는 메트릭 임계값 등 객관적 시그널)

## 스텝별 상세 계획

### Step 1 — {단계 제목}

1. (구체적 액션 — 파일 경로 또는 명령어 포함)
2. (구체적 액션)
3. (구체적 액션)

- **대상 파일**: `path/to/file`
- **검증**: (이 단계 완료 확인 방법)

### Step 2 — {단계 제목}

1. (구체적 액션)
2. (구체적 액션)

- **대상 파일**: `path/to/file`
- **검증**: (이 단계 완료 확인 방법)
```

- 완료 조건은 plan **전체**의 수용 기준이며, 스텝별 `검증`은 해당 단계 내부 확인용입니다 (역할이 다름)
- 항목은 검증 가능한 결과(명령어 출력, 리소스 상태, 메트릭 임계값, 산출물 존재 등) 중심으로 작성합니다
- 주관적 표현("잘 동작한다", "문제 없다") 지양, 객관적 시그널을 명시합니다

설계 과정에서 복수의 접근 방식이 존재할 때는 아래 형식으로 옵션을 비교합니다:

```
## 옵션 비교

### Option A: (옵션명)
- **장점**: (핵심 이점)
- **단점**: (트레이드오프/리스크)

### Option B: (옵션명)
- **장점**: (핵심 이점)
- **단점**: (트레이드오프/리스크)

**추천**: Option X — (선택 이유 한 줄)
```

- 옵션이 1개뿐이면 비교 섹션 생략
- 추천에는 반드시 선택 이유를 명시

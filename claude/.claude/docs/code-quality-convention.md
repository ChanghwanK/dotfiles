# Code Quality Convention

이 문서는 코드 작성, 수정, 리팩터링, 테스트 작성 시 사용하는 전역 품질 기준입니다.
`CLAUDE.md` 또는 `AGENTS.md`에는 짧은 진입점만 두고, 세부 판단은 이 문서를 참조합니다.

## 핵심 정의

좋은 코드는 단순히 짧거나 보기 좋은 코드가 아닙니다.
좋은 코드는 미래의 변경자가 의도를 이해하고, 영향 범위를 예측하고, 기대 동작을 검증할 수 있는 코드입니다.

가독성은 포맷만의 문제가 아니라 아래 3가지의 합입니다.

- **의도**: 이 코드가 왜 존재하고 어떤 정책을 표현하는지 드러나는가?
- **경계**: 어디까지가 이 코드의 책임이고, 어디를 바꾸면 위험한지 보이는가?
- **검증 가능성**: 기대 동작이 테스트, 타입, 검증, 에러 처리로 확인 가능한가?

## 코드 작업 기본 원칙

- 겉보기의 간결함보다 유지보수 가능한 명확성을 우선합니다.
- 기존 코드베이스의 패턴을 먼저 따르고, 벗어날 때는 이유를 명시합니다.
- 변경 범위는 요청을 해결하는 가장 작은 안전한 경계로 제한합니다.
- 새 추상화는 실제 복잡도를 줄이거나 책임 경계를 명확히 할 때만 도입합니다.
- 테스트 보강은 변경 위험도와 blast radius에 맞춰 결정합니다.

## 1. 의도를 드러내는 코드

- 변수명, 함수명, 타입명, 상수명은 도메인 의미를 담아야 합니다.
- `data`, `result`, `handler`, `process` 같은 넓은 이름은 작은 스코프에서 의미가 명확할 때만 사용합니다.
- magic number, magic string, 상태값은 named constant 또는 enum으로 표현합니다.
- 조건문은 가능한 한 비즈니스 규칙이 읽히게 작성합니다.
- 짧지만 추론이 필요한 코드보다, 조금 길어도 판단 근거가 보이는 코드를 선호합니다.

```ts
// 부족한 코드
if (user.type === 2 && score > 80) {
  upgrade(user)
}

// 더 좋은 코드
if (user.isPaidMember && score >= PREMIUM_ELIGIBILITY_SCORE) {
  promoteToPremiumCandidate(user)
}
```

## 2. 책임 경계가 명확한 구조

- 함수 하나가 여러 책임을 동시에 갖지 않게 합니다.
- 도메인 로직, I/O, 외부 API 호출, 포맷 변환, 영속성 처리는 가능한 한 분리합니다.
- 추상화는 내부를 숨기기 위해서가 아니라 변경 경계를 명확히 하기 위해 사용합니다.
- 너무 잘게 나눠 실행 흐름이 흐려지는 구조는 피합니다.
- 기준은 함수 길이가 아니라 "변경 시 영향 범위를 예측할 수 있는가"입니다.

## 3. 주석 기준

- 주석은 코드가 이미 말하는 "무엇"을 반복하지 않습니다.
- 주석은 코드에 드러나지 않는 "왜"를 설명합니다.
- 비즈니스 예외, 운영 제약, 과거 장애에서 나온 우회, 트레이드오프는 주석으로 남길 수 있습니다.
- 코드와 맞지 않는 주석은 없는 것보다 위험하므로, 변경 시 함께 수정하거나 제거합니다.

```ts
// 나쁜 주석
// user를 필터링한다
const users = allUsers.filter(user => user.active)

// 좋은 주석
// 비활성 계정은 결제 상태가 남아 있어도 캠페인 대상에서 제외한다.
// CS에서 수동 복구하는 계정이 포함되면 중복 쿠폰이 발급될 수 있다.
const eligibleUsers = allUsers.filter(user => user.active)
```

## 4. 테스트 코드 기준

좋은 테스트 코드는 기대 동작을 읽을 수 있는 문서이자 변경 안전장치입니다.

- 테스트 이름은 구현 방식이 아니라 정책 또는 사용자에게 관찰되는 동작을 설명합니다.
- Given / When / Then 흐름이 자연스럽게 읽히도록 작성합니다.
- 내부 구현 디테일보다 외부에서 관찰 가능한 결과를 우선 검증합니다.
- 정상 케이스뿐 아니라 경계값, 실패 케이스, 권한 없음, 중복 요청, 이미 처리된 상태, 시간/타임존, 외부 의존성 실패를 필요에 따라 포함합니다.
- 테스트 데이터는 의도를 드러내야 하며, 의미 없는 랜덤 값이나 과도하게 현실적인 fixture는 피합니다.
- 하나의 테스트는 보통 하나의 이유로 실패해야 합니다.
- 테스트는 결정적이어야 합니다. 실제 시간, 랜덤 값, 네트워크, 공유 상태, 실행 순서에 의존하지 않게 제어합니다.
- 테스트 helper는 반복 노이즈를 줄이되, 시나리오의 의미를 숨기지 않아야 합니다.

```ts
it("does not issue coupons to inactive paid users", () => {
  const user = createUser({ active: false, subscription: "paid" })

  const result = issueCoupon(user)

  expect(result).toEqual({ issued: false, reason: "INACTIVE_USER" })
})
```

## 5. 추상화 기준

- 중복이 보인다는 이유만으로 추상화하지 않습니다.
- 같은 이유로 함께 변경되는 코드라면 추상화 후보입니다.
- 서로 다른 정책이 우연히 비슷한 모양을 가진 경우에는 명시적 중복이 더 안전할 수 있습니다.
- generic helper에 flag, mode, 예외 조건이 계속 늘어나면 추상화가 잘못된 신호입니다.
- 호출자가 내부 주의사항을 알아야만 안전하게 쓸 수 있다면 좋은 추상화가 아닙니다.

## 6. 변경 안전성 기준

- 타입, 인터페이스, validation으로 잘못된 사용을 가능한 한 앞에서 막습니다.
- 에러 메시지는 운영자나 개발자가 원인을 좁힐 수 있게 작성합니다.
- 중요한 도메인 규칙은 한 곳에서 확인 가능해야 합니다.
- 리팩터링은 기존 동작을 보존해야 하며, 보존 여부를 테스트 또는 명확한 검증으로 확인합니다.
- 변경 후에는 영향 범위, 실패 시나리오, 롤백 방법을 설명할 수 있어야 합니다.

## 작업 전 체크리스트

코드 변경 전 다음을 확인합니다.

- 이 변경은 증상 해결인가, 근본 원인 해결인가?
- 기존 패턴을 따르는가? 다르면 왜 다른가?
- 변경 경계가 요청 범위에 맞게 작게 잡혔는가?
- 테스트 또는 검증으로 기대 동작을 확인할 수 있는가?
- 실패했을 때 원복 방법이 명확한가?

## 작업 후 체크리스트

코드 변경 후 다음을 확인합니다.

- 이름과 구조만으로 의도가 읽히는가?
- 책임 경계가 흐려진 부분은 없는가?
- 주석은 필요한 이유만 설명하는가?
- 테스트가 정책, 경계값, 실패 조건 중 변경 위험에 맞는 부분을 보호하는가?
- 새 추상화가 실제 복잡도를 줄였는가?
